# DailyForge 技术选型方案

## 1. 文档目标

本文档用于确定 DailyForge 在 MVP 阶段的技术选型、架构边界和配套工程方案，目标是在保证后续可扩展性的前提下，优先降低个人项目的开发成本和维护复杂度。

## 2. 结论摘要

当前这套技术选型整体是合适的，尤其适合以下目标：

- 前端使用现代 TypeScript 技术栈，保证开发效率和类型安全
- 后端使用 Java + Spring 体系，保证工程规范、扩展性和生态成熟度
- 数据层使用 MySQL + Redis，满足用户、计划、打卡、统计等核心业务
- 部署阶段先采用 Docker Compose，便于本地开发和单机部署

但有一个关键优化建议：

- MVP 阶段建议采用“前后端分离 + 模块化单体后端”架构
- 不建议在第一版就引入完整微服务体系

原因是 DailyForge 当前 MVP 的业务复杂度还不足以支撑微服务拆分，过早引入 Gateway、Nacos、配置中心、服务注册发现等设施，会明显提高开发、调试、部署和排障成本。

因此推荐的落地策略是：

- 第一阶段：单体 Spring Boot 后端 + Vite React 前端
- 第二阶段：当用户规模、团队规模或业务边界变复杂后，再逐步引入 Spring Cloud 体系

## 3. 技术选型总览

### 3.1 前端

- 语言：TypeScript 5.7
- UI 框架：React 19.0
- 构建工具：Vite 6.3
- CSS 框架：Tailwind CSS 4.2
- 包管理：pnpm 9.15

### 3.2 后端

- 语言：Java 21
- 框架：Spring Boot 3.2.5
- 构建工具：Maven
- 微服务框架：Spring Cloud 2023.0.3
- Alibaba 生态：Spring Cloud Alibaba 2023.0.1.3
- 可选微服务设施：Spring Cloud Gateway、Nacos、Nacos Config、Spring Cloud LoadBalancer、Redisson、OkHttp

### 3.3 数据层

- 关系数据库：MySQL 8.0
- ORM：MyBatis-Plus 3.5.5
- 缓存：Redis 7

### 3.4 可观测性

- 日志框架：SLF4J + Logback
- 日志格式：结构化 JSON
- Trace 标识：W3C Trace Context 风格字段，例如 `trace_id`、`span_id`

### 3.5 部署与容器化

- 部署方式：Docker Compose

## 4. 是否合适

### 4.1 前端选型判断

这套前端技术栈是合适的，优点如下：

- `TypeScript + React` 适合中长期维护和组件化开发
- `Vite` 启动快、构建快，适合个人项目和中小型前端应用
- `Tailwind CSS 4` 能较快搭建高一致性的界面
- `pnpm` 安装快、磁盘占用更低，适合后续扩展为多包仓库

需要注意的点：

- Tailwind CSS 4 与 Vite 的官方推荐方式是使用 `@tailwindcss/vite` 插件，而不是沿用旧版 PostCSS 配置习惯
- React 19 已经稳定，但应尽量避免一开始就使用过多新能力，优先使用成熟稳定的组件和状态管理写法

### 4.2 后端选型判断

Java 21 + Spring Boot 3.2.x 非常适合 DailyForge 这类中后台业务系统，优点如下：

- 语言和框架成熟
- 生态完善
- 适合做账户、计划、训练记录、统计分析等典型 CRUD + 规则判断类业务
- 后续无论继续单体演进还是拆服务，迁移路径都比较清晰

MyBatis-Plus 也适合当前阶段，因为：

- 对表结构明确、查询逻辑可控的业务非常高效
- 对个人项目来说比 JPA 更直接
- 后续在复杂查询、统计聚合、手写 SQL 上更灵活

### 4.3 微服务选型判断

你的微服务栈并不是“不合适”，但它更适合第二阶段，而不是 MVP 第一阶段。

MVP 阶段如果直接上：

- Spring Cloud Gateway
- Nacos
- Nacos Config
- LoadBalancer
- 多服务拆分

会带来以下问题：

- 本地开发环境复杂度明显提高
- 配置管理、端口管理、服务依赖关系更繁琐
- 接口联调、日志排查、链路定位成本更高
- 对当前业务规模来说收益不明显

因此推荐：

- 先保留 Spring Cloud 和 Spring Cloud Alibaba 为“预备技术栈”
- 第一版只实现单个后端服务
- 等到后续确实出现用户服务、计划服务、记录服务独立演进需求时再拆

## 5. 兼容性与版本注意事项

### 5.1 前端版本组合

这组版本方向上没有明显问题：

- React 19 已稳定发布
- Tailwind 官方文档提供了基于 Vite 的安装方式
- Vite 适合作为 React 项目的构建工具

建议补充一个运行时版本约束：

- Node.js：建议固定为 LTS 版本，例如 Node.js 22 LTS

原因是前端实际是否稳定，不只取决于 React / Vite / Tailwind 本身，还取决于 Node 运行时版本。

### 5.2 Spring Cloud / Spring Cloud Alibaba 组合

这一块需要特别谨慎。

根据 Spring Cloud 官方支持矩阵，`Spring Cloud 2023.0` 兼容 `Spring Boot 3.2.x`，也兼容 `3.3.x`。  
根据 Spring Cloud Alibaba 官方 2023.x 文档页，公开展示的适配关系中，`2023.0.1.0` 对应 `Spring Cloud 2023.0.1` 和 `Spring Boot 3.2.4`。  
同时，Spring Cloud Alibaba 官方 GitHub Release 页面可以看到 `2023.0.1.3` 版本已经存在。

基于这些信息，可以做出一个谨慎结论：

- 你的 `Spring Boot 3.2.5 + Spring Cloud 2023.0.3` 主线方向是合理的
- `Spring Cloud Alibaba 2023.0.1.3` 很可能可用，但在正式落库到 `pom.xml` 前，应再次核对该版本 release note 或 BOM 说明

因此本项目建议：

- 如果 MVP 阶段不依赖 Nacos、SCA 注册发现、SCA 配置中心，就先不要引入 Spring Cloud Alibaba
- 如果必须使用 Nacos，再单独锁定一组已验证通过的 BOM 组合

## 6. 推荐架构方案

### 6.1 MVP 阶段推荐架构

推荐采用以下结构：

- 前端：独立 React SPA
- 后端：单体 Spring Boot API 服务
- 数据库：MySQL
- 缓存：Redis
- 部署：Docker Compose

前后端职责建议如下：

- 前端负责页面渲染、交互、表单、图表展示
- 后端负责鉴权、计划生成规则、训练记录、饮食建议、统计聚合、AI 调用代理

### 6.2 后续扩展架构

如果项目进入第二阶段，再考虑以下方向：

- 增加 API Gateway
- 接入 Nacos 服务注册与配置中心
- 按领域拆分用户服务、计划服务、训练记录服务、统计服务
- 增加异步消息处理

## 7. 建议补充的技术与配套

以下是当前清单里建议补齐的部分。

### 7.1 前端建议补充

- 路由：React Router
- 服务端状态管理：TanStack Query
- 表单处理：React Hook Form
- 表单校验：Zod
- 图表库：ECharts 或 Recharts
- 时间处理：dayjs
- 图标库：Lucide React

说明：

- 不建议一开始就引入过重的全局状态管理
- 如果只是用户信息、计划草稿、UI 状态，可以优先使用 React 自身状态，必要时再补 Zustand

### 7.2 后端建议补充

- Web：Spring Web
- 参数校验：Spring Validation
- 安全：Spring Security
- 鉴权：JWT
- API 文档：springdoc-openapi
- 数据库初始化：SQL 脚本手动执行
- 对象映射：MapStruct
- 连接池：HikariCP
- 监控：Spring Boot Actuator

说明：

- Spring Security 和 JWT 是必须补齐的，用户数据涉及体重、训练目标、饮食建议，不能弱化鉴权
- 当前阶段可先使用版本化 SQL 脚本手动初始化数据库，后续再视兼容性与团队习惯决定是否重新引入迁移框架
- MapStruct 可以降低 DTO / DO / VO 转换成本

### 7.3 可观测性建议补充

- 指标采集：Micrometer
- 监控面板：Prometheus + Grafana
- 链路追踪：Micrometer Tracing 或 OpenTelemetry

说明：

- 你已经定义了结构化 JSON 日志，这很好
- 但如果要真正打通 `trace_id` 和 `span_id`，仅有 Logback 不够，还需要 tracing 组件负责生成和传递链路上下文

### 7.4 测试建议补充

- 前端单测：Vitest
- 前端组件测试：Testing Library
- 后端单测：JUnit 5
- 后端集成测试：Spring Boot Test
- 基础设施测试：Testcontainers

说明：

- MySQL、Redis 相关集成测试很适合用 Testcontainers
- 对个人项目来说，不必追求极高覆盖率，但关键业务流程必须有测试

### 7.5 AI 能力接入建议

DailyForge 后续如果接入 AI，建议补充一层独立的 AI 适配模块，而不是把调用逻辑直接散落在业务代码中。

建议至少拆出以下职责：

- Prompt 模板管理
- 模型调用封装
- 请求与响应日志记录
- 限流与重试
- 降级兜底

HTTP 客户端方面建议二选一：

- 外部 AI API 调用走 `OkHttp`
- 或统一走 Spring 的 `WebClient`

不建议同一项目同时混用多个主要 HTTP 客户端，避免拦截器、超时、日志、重试策略分散。

## 8. 部署建议

Docker Compose 对 MVP 阶段是合适的，但建议把部署清单定义完整。

建议至少包含：

- `frontend`
- `backend`
- `mysql`
- `redis`
- `nginx`，可选但推荐

同时建议补齐：

- 数据卷挂载
- 环境变量管理
- 基础健康检查
- 时区统一配置
- 日志落盘策略
- 数据库备份策略

## 9. 最终推荐版本策略

### 9.1 MVP 实施版本

推荐你在 MVP 阶段采用以下落地方案：

- 前端：TypeScript + React + Vite + Tailwind CSS + pnpm
- 后端：Java 21 + Spring Boot 3.2.5 + Maven
- 数据层：MySQL 8.0 + Redis 7 + MyBatis-Plus
- 鉴权：Spring Security + JWT
- 文档：springdoc-openapi
- 数据初始化：版本化 SQL 脚本
- 可观测性：SLF4J + Logback + Micrometer + Actuator
- 部署：Docker Compose

### 9.2 MVP 暂缓引入

以下能力建议暂缓到第二阶段：

- Spring Cloud Gateway
- Nacos
- Nacos Config
- Spring Cloud LoadBalancer
- 全面微服务拆分
- Redisson 的分布式高级能力

其中 Redis 可以保留，但 Redisson 建议按需引入，不要预装。

## 10. 结论

你的技术路线总体是正确的，属于“前端现代化 + 后端企业级 + 数据层稳妥”的组合，能支撑 DailyForge 从 MVP 做到后续扩展。

最需要优化的不是换框架，而是控制复杂度：

- MVP 先做模块化单体
- 微服务相关基础设施延后
- 提前补齐鉴权、数据库迁移、测试、监控这些容易被忽略但非常重要的配套

如果按这个思路推进，这套技术栈是可以落地的，而且后续演进空间也足够。
