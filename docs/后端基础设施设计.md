# DailyForge 后端基础设施设计

## 1. 文档目标

本文档用于说明 DailyForge 后端基础设施层的最小实现方案，帮助后续开发和改造时快速理解以下内容：

- 统一返回体设计
- 错误码设计
- 业务异常与全局异常处理机制
- Spring Security 最小安全配置
- OpenAPI 基础配置

本文档只覆盖当前第一阶段已经落地的基础设施，不展开 JWT 认证、权限注解或复杂审计能力。

## 2. 目录与类职责

当前基础设施相关类分布如下：

- `com.dailyforge.common.ApiResponse`
- `com.dailyforge.common.ErrorCode`
- `com.dailyforge.common.BusinessException`
- `com.dailyforge.common.handler.GlobalExceptionHandler`
- `com.dailyforge.config.SecurityConfig`
- `com.dailyforge.config.OpenApiConfig`

各类职责如下：

- `ApiResponse`
  - 统一 API 返回结构
- `ErrorCode`
  - 定义全局错误码、默认消息、对应 HTTP 状态
- `BusinessException`
  - 业务层主动抛出的运行时异常
- `GlobalExceptionHandler`
  - 将异常统一转换为标准 API 响应
- `SecurityConfig`
  - 定义第一阶段的 Spring Security 安全边界
- `OpenApiConfig`
  - 定义 Swagger/OpenAPI 的基础信息与 Bearer Token 安全方案

## 3. 统一返回体设计

### 3.1 返回结构

统一返回结构为：

```json
{
  "code": "SUCCESS",
  "message": "ok",
  "data": {}
}
```

字段含义如下：

- `code`
  - 字符串错误码或成功码
- `message`
  - 面向前端和日志阅读的简短说明
- `data`
  - 实际业务数据

设计原则：

- 成功与失败都返回同一种结构
- 不额外增加 `success` 布尔字段
- 使用字符串码值，避免纯数字码在语义上不直观

### 3.2 `ApiResponse<T>` 类

文件：

- [ApiResponse.java](</D:\Computer Science\DailyForge\backend\src\main\java\com\dailyforge\common\ApiResponse.java>)

类作用：

- 统一封装接口返回值
- 提供静态工厂方法，降低 Controller 和异常处理器的样板代码

结构：

- `code: String`
- `message: String`
- `data: T`

采用 `record` 的原因：

- 结构固定
- 代码量少
- 适合做不可变响应对象

### 3.3 `ApiResponse` 工厂方法

#### `success(T data)`

作用：

- 返回带数据的成功响应

参数：

- `data`
  - 成功时要返回的业务数据

返回值：

- `ApiResponse<T>`

返回内容：

- `code = SUCCESS`
- `message = ok`
- `data = 传入数据`

#### `success()`

作用：

- 返回不带业务数据的成功响应

参数：

- 无

返回值：

- `ApiResponse<Void>`

#### `failure(ErrorCode errorCode)`

作用：

- 按错误码默认文案构造失败响应

参数：

- `errorCode`
  - 错误码枚举

返回值：

- `ApiResponse<T>`

#### `failure(ErrorCode errorCode, String message)`

作用：

- 按错误码和自定义消息构造失败响应

参数：

- `errorCode`
  - 错误码枚举
- `message`
  - 自定义错误文案

返回值：

- `ApiResponse<T>`

## 4. 错误码设计

### 4.1 设计原则

错误码当前采用枚举集中维护，每个错误码包含三部分：

- `code`
- `defaultMessage`
- `httpStatus`

这样可以保证：

- 响应结构统一
- HTTP 状态语义明确
- 错误码含义在后端集中管理

### 4.2 `ErrorCode` 类

文件：

- [ErrorCode.java](</D:\Computer Science\DailyForge\backend\src\main\java\com\dailyforge\common\ErrorCode.java>)

字段：

- `code: String`
- `defaultMessage: String`
- `httpStatus: HttpStatus`

### 4.3 当前错误码枚举

#### `SUCCESS`

含义：

- 请求成功

默认文案：

- `ok`

HTTP 状态：

- `200 OK`

#### `INVALID_ARGUMENT`

含义：

- 请求参数非法、缺失或校验不通过

默认文案：

- `request arguments are invalid`

HTTP 状态：

- `400 BAD_REQUEST`

#### `UNAUTHORIZED`

含义：

- 需要认证但当前未登录或凭证无效

默认文案：

- `authentication is required`

HTTP 状态：

- `401 UNAUTHORIZED`

#### `FORBIDDEN`

含义：

- 已认证但没有对应权限

默认文案：

- `permission denied`

HTTP 状态：

- `403 FORBIDDEN`

#### `RESOURCE_NOT_FOUND`

含义：

- 请求的资源不存在

默认文案：

- `resource not found`

HTTP 状态：

- `404 NOT_FOUND`

#### `INTERNAL_SERVER_ERROR`

含义：

- 未捕获异常或未知服务端错误

默认文案：

- `internal server error`

HTTP 状态：

- `500 INTERNAL_SERVER_ERROR`

## 5. 异常体系设计

### 5.1 整体思路

第一阶段异常处理遵循以下规则：

- 业务逻辑主动失败：抛 `BusinessException`
- 参数校验失败：由 Spring Validation 异常接管
- 其余异常：统一兜底

不推荐在 Controller 内部手写大量 `try/catch`。

### 5.2 `BusinessException`

文件：

- [BusinessException.java](</D:\Computer Science\DailyForge\backend\src\main\java\com\dailyforge\common\BusinessException.java>)

类作用：

- 表示可预期的业务异常

字段：

- `errorCode: ErrorCode`

构造方法：

#### `BusinessException(ErrorCode errorCode)`

作用：

- 按错误码默认文案抛出业务异常

参数：

- `errorCode`
  - 业务错误码

#### `BusinessException(ErrorCode errorCode, String message)`

作用：

- 按错误码和自定义文案抛出业务异常

参数：

- `errorCode`
  - 业务错误码
- `message`
  - 自定义错误消息

使用方式示例：

```java
throw new BusinessException(ErrorCode.RESOURCE_NOT_FOUND);
```

或：

```java
throw new BusinessException(ErrorCode.FORBIDDEN, "current user cannot access this resource");
```

### 5.3 全局异常处理器

文件：

- [GlobalExceptionHandler.java](</D:\Computer Science\DailyForge\backend\src\main\java\com\dailyforge\common\handler\GlobalExceptionHandler.java>)

类作用：

- 统一拦截异常并转换为标准 API 返回结构

当前处理的异常类型如下。

#### `BusinessException`

处理逻辑：

- 读取异常内的 `ErrorCode`
- 按 `ErrorCode` 对应的 HTTP 状态返回
- 响应体结构为 `ApiResponse.failure(...)`

#### `MethodArgumentNotValidException`

触发场景：

- `@Valid` 请求体校验失败

处理逻辑：

- 收集所有字段错误
- 拼接为单条字符串消息
- 返回 `INVALID_ARGUMENT`

#### `ConstraintViolationException`

触发场景：

- 路径参数、查询参数等约束校验失败

处理逻辑：

- 收集约束违规信息
- 返回 `INVALID_ARGUMENT`

#### `Exception`

触发场景：

- 未被上面异常处理器覆盖的其他异常

处理逻辑：

- 记录 error 级别日志
- 返回 `INTERNAL_SERVER_ERROR`

### 5.4 `GlobalExceptionHandler` 内部方法

#### `handleBusinessException(BusinessException exception)`

作用：

- 处理业务异常

参数：

- `exception`
  - 业务异常对象

返回值：

- `ResponseEntity<ApiResponse<Void>>`

#### `handleMethodArgumentNotValidException(MethodArgumentNotValidException exception)`

作用：

- 处理请求体验证失败

参数：

- `exception`
  - Spring 请求体校验异常

返回值：

- `ResponseEntity<ApiResponse<Void>>`

#### `handleConstraintViolationException(ConstraintViolationException exception)`

作用：

- 处理参数级约束异常

参数：

- `exception`
  - 约束违规异常

返回值：

- `ResponseEntity<ApiResponse<Void>>`

#### `handleException(Exception exception)`

作用：

- 处理未预料到的异常

参数：

- `exception`
  - 任意异常

返回值：

- `ResponseEntity<ApiResponse<Void>>`

#### `formatFieldError(FieldError fieldError)`

作用：

- 将字段错误格式化成易读字符串

参数：

- `fieldError`
  - Spring 字段错误对象

返回值：

- `String`

## 6. Spring Security 基础配置

### 6.1 设计目标

第一阶段安全配置只做最小可用边界，不在这一轮实现完整 JWT 鉴权。

目标如下：

- 关闭不需要的默认安全能力
- 明确哪些接口默认放行
- 其余接口默认要求认证
- 为下一步接入 JWT 留出稳定入口

### 6.2 `SecurityConfig`

文件：

- [SecurityConfig.java](</D:\Computer Science\DailyForge\backend\src\main\java\com\dailyforge\config\SecurityConfig.java>)

类作用：

- 定义 Spring Security 过滤链
- 定义密码编码器

### 6.3 当前安全策略

当前配置包括：

- 关闭 `CSRF`
- 关闭表单登录
- 关闭 HTTP Basic
- 使用 `STATELESS` 会话策略
- 开启默认 CORS 支持

默认放行路径：

- `/auth/**`
- `/docs/**`
- `/actuator/health`
- `/error`

其余所有请求：

- 默认要求已认证

### 6.4 `SecurityConfig` 方法

#### `securityFilterChain(HttpSecurity http)`

作用：

- 构建整个 Spring Security 过滤链

参数：

- `http`
  - Spring 提供的 `HttpSecurity`

返回值：

- `SecurityFilterChain`

当前未做的事情：

- 未接入 JWT 过滤器
- 未接入 `UserDetailsService`
- 未接入权限注解

#### `passwordEncoder()`

作用：

- 提供密码编码器 Bean

返回值：

- `PasswordEncoder`

当前实现：

- `BCryptPasswordEncoder`

用途：

- 注册时加密密码
- 登录时校验密码

## 7. OpenAPI 基础配置

### 7.1 设计目标

OpenAPI 第一阶段只做三件事：

- 提供基础 API 文档信息
- 开启 Swagger UI
- 提前定义 Bearer Token 安全方案

### 7.2 `OpenApiConfig`

文件：

- [OpenApiConfig.java](</D:\Computer Science\DailyForge\backend\src\main\java\com\dailyforge\config\OpenApiConfig.java>)

类作用：

- 定义 OpenAPI 文档元信息
- 定义 Bearer Token 安全方案

### 7.3 `OpenApiConfig` 方法

#### `dailyForgeOpenApi()`

作用：

- 构造 `OpenAPI` Bean

返回值：

- `OpenAPI`

当前配置内容：

- 标题：`DailyForge API`
- 版本：`v1`
- 描述：`Core backend API for DailyForge.`
- 安全方案：`bearerAuth`

Bearer Token 安全方案定义：

- `type = HTTP`
- `scheme = bearer`
- `bearerFormat = JWT`

说明：

- 当前文档层已经支持 Bearer Token 描述
- 真正的 JWT 认证逻辑会在后续认证模块中接入

## 8. 当前基础设施建设范围

这一轮已经完成的基础设施包括：

- 统一响应结构
- 错误码枚举
- 业务异常
- 全局异常处理
- Spring Security 最小边界
- OpenAPI 基础配置

这一轮明确未实现的内容包括：

- JWT 生成与校验
- 自定义认证过滤器
- 用户认证上下文
- 细粒度权限控制
- 审计日志
- 国际化错误消息

## 9. 后续建议顺序

在当前基础设施之上，建议按以下顺序继续推进：

1. `auth` 模块
   - 注册
   - 登录
   - 密码加密校验
   - JWT 发放

2. `user/profile` 模块
   - 当前用户信息
   - 用户基础档案维护

3. 数据访问层基础配置
   - MyBatis-Plus 配置
   - Mapper 扫描
   - 基础实体约定

4. JWT 认证接入
   - 过滤器
   - Token 解析
   - 当前用户上下文

## 10. 结论

当前这套基础设施的目标不是一次性做完整，而是先把后续模块开发必须依赖的公共约束立住：

- 响应怎么返回
- 错误怎么表达
- 异常怎么统一处理
- 哪些接口默认放行
- 文档怎么调试

这样后续开发 `auth`、`user`、`profile`、`plan` 等模块时，就不需要重复设计底层规范，也能显著降低后期改造成本。
