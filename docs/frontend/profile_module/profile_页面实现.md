## 2. `ProfilePage`

建议文件：

- `ProfilePage.tsx`

建议路由：

- `/profile`

### 2.1 页面目标

这是用户日常进入和维护个人资料的主页面。

### 2.2 页面结构

建议结构如下：

1. 页面标题区
2. 资料状态提示区
3. Tab 切换区
4. 当前 Tab 对应内容区

### 2.3 标题区

建议包含：

- 页面标题：`个人资料`
- 辅助说明：说明资料对 AI 建议与训练记录的价值

### 2.4 资料状态提示区

建议使用 `CompletionSummaryBanner` 展示：

- 基础档案是否可用于 AI
- 当前是否已有体重记录
- 哪些关键字段仍然缺失

目标不是制造压力，而是提供明确状态反馈。

### 2.5 Tab 切换区

包含两个 Tab：

- `基础档案`
- `身体指标`

建议默认进入：

- `基础档案`

如果用户是从 AI 缺失资料引导页跳转过来，也可以根据来源默认切到更需要的 Tab。

---

## 3. `基础档案` Tab

### 3.1 页面组成

建议拆为：

1. 左侧或顶部：说明卡片
2. 主体：`BasicProfileForm`
3. 右侧或下方：当前摘要卡片

### 3.2 表单字段

字段包括：

- 性别
- 出生日期
- 身高
- 训练目标
- 训练经验
- 伤病/注意事项

### 3.3 控件建议

| 字段 | 建议控件 |
|------|------|
| 性别 | 单选或下拉 |
| 出生日期 | 日期选择器 |
| 身高 | 数字输入框 |
| 训练目标 | 单选卡片或下拉 |
| 训练经验 | 单选卡片 |
| 伤病/注意事项 | 多行文本框 |

### 3.4 提交行为

点击“保存基础档案”后：

1. 前端执行字段格式校验
2. 调用 `PUT /api/profile/basic`
3. 成功后刷新基础档案与完成度摘要
4. 展示成功提示

### 3.5 页面提示

应明确说明：

- 这些字段不是强制完整填写门槛
- 但越完整，AI 建议越准确

---

## 4. `身体指标` Tab

### 4.1 页面组成

建议拆成三个纵向区块：

1. 当前身体状态摘要
2. 身体指标录入表单
3. 最近历史记录列表

### 4.2 当前身体状态摘要区

建议展示：

- 当前体重
- 当前体脂
- 当前 BMI
- 当前骨骼肌率
- 当前水分
- 当前基础代谢
- 当前腰围 / 臀围 / 腰臀比
- 当前身体年龄
- 当前体型
- 快照更新时间

显示原则：

- 有值就展示
- 无值显示 `--`

### 4.3 身体指标录入区

建议使用完整表单而非弹窗，降低理解成本。

字段包括：

- 记录日期
- 体重
- 体脂率
- BMI
- 骨骼肌率
- 身体水分
- 基础代谢
- 腰围
- 臀围
- 腰臀比
- 身体年龄
- 体型
- 备注

### 4.4 表单说明

需要明确提示：

- `记录日期` 必填
- 其余字段可按掌握情况填写
- 至少填写一个有效身体指标，不能只写备注

### 4.5 提交行为

点击“新增记录”后：

1. 前端做轻量校验
2. 调用 `POST /api/profile/body-metrics`
3. 成功后重置表单或保留记录日期
4. 刷新摘要、列表、完成度和基础档案摘要

---

## 5. 历史记录列表区

### 5.1 展示目标

第一版不做复杂图表，只展示最近历史记录。

### 5.2 列表字段建议

每条记录建议展示：

- 记录日期
- 体重
- 体脂率
- BMI
- 身体年龄
- 体型
- 备注
- 是否为最新记录

### 5.3 `isLatest` 的呈现

对 `isLatest=true` 的项建议展示：

- `最新记录` 标签
- 同时提供“删除最新记录”按钮

### 5.4 分页方式

第一版建议传统分页或“加载更多”都可以。  
如果历史不会很长，先做简单分页即可。

### 5.5 删除交互

点击删除最新记录后：

1. 打开确认弹窗
2. 用户确认后调用 `DELETE /api/profile/body-metrics/latest`
3. 成功后刷新摘要、列表、完成度和基础档案摘要

## 6. `ProfileOnboardingPage`

建议文件：

- `ProfileOnboardingPage.tsx`

建议路由：

- `/profile/onboarding`

### 6.1 页面目标

注册成功后给用户一次轻量欢迎和资料引导。

### 6.2 页面结构

建议两步式布局：

#### 第一步

- 欢迎说明
- 基础档案表单
- 操作按钮：`跳过`、`保存并继续`

#### 第二步

- 身体指标录入表单
- 操作按钮：`上一步`、`跳过并进入应用`、`保存并完成`

### 6.3 页面交互原则

- 允许跳过
- 跳过不阻塞后续功能
- 但要明确告诉用户：后续可以在个人资料页继续补充

### 6.4 完成后的跳转

建议跳转到：

- `/app`

## 7. `ProfileAiCompletionPage`

建议文件：

- `ProfileAiCompletionPage.tsx`

建议路由：

- `/profile/ai-completion`

### 7.1 页面目标

当用户准备使用 AI 计划、AI 饮食、AI 总结，但资料不足时，给出一次场景化补录入口。

### 7.2 与 onboarding 的区别

它与 onboarding 可以共用大部分表单，但页面语义不同：

- onboarding：欢迎与首次引导
- ai-completion：为了获得更准确 AI 结果而补充资料

### 7.3 页面结构建议

1. 顶部说明卡片
2. 缺失字段清单
3. 两步补录区域
4. 完成后返回来源功能页

### 7.4 跳转回来源页

建议在 query 中带来源，例如：

`/profile/ai-completion?scene=ai-plan&redirect=/ai/plan`

页面完成或跳过后：

- 回到 `redirect`

## 8. 前端与后端交互刷新矩阵

| 操作 | 刷新 basic | 刷新 summary | 刷新 snapshot | 刷新 history |
|------|------|------|------|------|
| 进入 `ProfilePage` | 是 | 是 | 是 | 是 |
| 保存基础档案 | 是 | 是 | 否 | 否 |
| 新增身体指标 | 是 | 是 | 是 | 是 |
| 删除最新身体指标 | 是 | 是 | 是 | 是 |

## 9. 推荐 UI 呈现风格

延续当前项目前端风格，建议：

- 使用深色背景与暖色高亮
- 表单区域使用半透明卡片
- 当前身体摘要以卡片阵列展示
- Tab 不要用默认浏览器样式

建议避免把页面做成传统设置页，因为 `profile` 对 DailyForge 是“高价值输入页”，不是低优先级附属页。

---

## 10. 已知实现风险

### 10.1 身体指标字段较多

如果把所有字段平铺到一个表单中，移动端会显得很长。建议至少做合理分组，例如：

- 基础指标
- 体脂与代谢
- 围度与体型
- 备注

### 10.2 查询与刷新点较多

`profile` 页对多个接口有联动刷新需求。若不用数据请求库，页面内部请求编排会变复杂，代码容易膨胀。

### 10.3 AI 引导页与首次引导页容易重复

建议复用组件和步骤结构，避免重复维护两套表单。
