## 1. 文档目标

本文档用于把“模板编辑器中的动作添加与配置体验优化”整理成一份可以直接交给前端 agent 的实施清单。

本次改造有两条主线：

1. 对接新版 `exercise` 查询接口，重构“添加动作”为弹窗选择器
2. 优化当前动作编辑器的交互效率，减少重复操作和视觉负担

---

## 2. 已确认的产品决策

以下结论已经确认，不需要再讨论：

1. 去掉动作卡片拖拽，保留上移 / 下移按钮。
2. “新增一组”默认复制上一组内容。
3. 动作备注改成按需展开，不默认展示输入框。
4. “添加动作”改为弹窗选择器，不再先创建空白动作卡片。
5. 弹窗左侧先使用粗粒度分类：
   - 胸
   - 背
   - 肩
   - 腿
   - 手臂
   - 核心
   - 有氧
6. 细分肌肉作为二级筛选补充能力，不要求一上来把所有细分肌肉直接铺在主界面。
7. 用户选中动作后再真正插入动作卡片。
8. 动作卡片中的搜索栏删除，改成“更换动作”按钮。

## 5. 具体改造清单

## 5.1 动作选择器弹窗

### 目标

把当前“先加空卡片，再在卡片内搜索”的流程，改成“先在弹窗里选动作，再生成动作卡片”。

### 目标交互

用户操作流程：

1. 点击“添加动作”
2. 打开动作选择器弹窗
3. 默认选中左侧第一个一级分类
4. 右侧自动加载该分类下的动作列表
5. 顶部搜索框可基于当前分类做关键词搜索
6. 点击某个动作后，关闭弹窗并插入动作卡片

### 弹窗内容建议

弹窗分区建议：

1. 顶部
   - 搜索框
   - 当前分类标题
2. 左侧
   - 一级分类列表
   - 后续可扩展二级肌肉筛选
3. 右侧
   - 动作列表
   - 动作标签
   - 选择按钮或整卡点击选择

### 组件改造建议

建议新增组件：

- `ExercisePickerDialog.tsx`
- 可选：`ExerciseCategorySidebar.tsx`
- 可选：`ExercisePickerList.tsx`

建议保留现有动作卡片编辑器，只把“选动作”流程剥离出去。

## 5.2 “添加动作”与“更换动作”流程

### 当前问题

现在 `onAddExercise(dayIndex)` 会直接插入一条空动作。

关键位置：

- [CycleTemplateEditor.tsx](/D:/Computer%20Science/DailyForge/frontend/src/features/cycle-template/components/CycleTemplateEditor.tsx)
- [useCycleTemplateEditor.ts](/D:/Computer%20Science/DailyForge/frontend/src/features/cycle-template/hooks/useCycleTemplateEditor.ts)

### 改造要求

1. 点击“添加动作”时，不要直接写入 `form.days[].exercises`
2. 先打开动作选择器弹窗
3. 只有用户在弹窗中确认选择动作后，才真正新增动作

### 状态设计建议

建议新增一个临时 UI 状态：

- 当前为哪一天添加动作
- 当前是否处于“更换动作”模式
- 如果是更换动作，需要记录目标 `exerciseLocalId`

建议新增两个动作：

1. `appendExerciseFromSystemOption(dayIndex, option)`
2. `replaceExerciseFromSystemOption(dayIndex, exerciseLocalId, option)`

说明：

- 不建议继续复用现在“先插空，再 select 替换”的模型
- 前端状态会更干净

## 5.6 “新增一组”默认复制上一组

这是本轮必须落实的优化点之一。

文件：

- [useCycleTemplateEditor.ts](/D:/Computer%20Science/DailyForge/frontend/src/features/cycle-template/hooks/useCycleTemplateEditor.ts)
- [ExerciseItemEditor.tsx](/D:/Computer%20Science/DailyForge/frontend/src/features/cycle-template/components/ExerciseItemEditor.tsx)

当前问题：

- `addItem(...)` 会基于 `createDefaultItemByStructureType(...)` 创建默认空白项

目标行为：

1. 第一组仍按默认空白模板创建
2. 第二组及以后默认复制上一组内容
3. 自动更新：
   - `itemIndex`
   - `sortOrder`
   - 本地 `localId`
   - 各 metric 的 `localId`
4. 建议清空上一组的备注，不强制复制备注

建议复制范围：

- 复制 `metricKey`
- 复制 `metricValueNumberText` 或对应数值输入内容
- 复制 `itemName` 可按实际决定

更稳妥的建议：

- 数值指标复制
- 执行项备注清空
- 执行项名称按新序号重建

说明：

- `single_segment` 保持现有只允许一个执行项的约束

## 5.8 弹窗默认加载策略

建议默认行为：

1. 打开弹窗时先请求 `filter-options`
2. 默认选中返回的第一个一级分类
3. 立即调用 `searchSystemExercises`，带上：
   - `categoryCode`
   - `page=1`
   - `pageSize=20`
4. 当用户输入关键词时：
   - 仍保留当前分类
   - 用 `categoryCode + keyword` 联合查询

说明：

- 这样初始界面不会是空白
- 用户不需要先输入关键词才看到可选动作

## 7. 验收清单

前端改造完成后，应满足以下条件：

1. 点击“添加动作”后，先打开弹窗，不会先插入空动作卡片。
2. 弹窗左侧可切换粗粒度分类，右侧动作列表会随之变化。
3. 顶部搜索框可与当前分类联合使用。
4. 选中动作后才真正插入卡片。
5. 动作卡片不再显示内嵌搜索区。
6. 动作卡片支持“更换动作”。
7. 动作卡片不再可拖拽，浏览器不会触发文本搜索 bug。
8. “新增一组”从第二组开始默认复制上一组内容。
9. 动作备注默认折叠，仅在有内容或用户主动添加时展开。
