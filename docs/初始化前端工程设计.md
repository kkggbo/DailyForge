## 1. 文档目标

本文档用于说明 DailyForge 前端工程第一版初始化时的设计思路、技术选型落地方式、目录结构、关键实现和当前边界，方便后续继续开发 `profile`、`cycle template`、`training session`、`stats` 等模块时快速接续。

本文档覆盖的内容包括：

- 前端初始化的目标与范围
- 已落地的技术选型与版本
- 工程目录结构
- 路由、鉴权、接口调用、样式体系的初始设计
- 已实现页面与功能边界
- 后续扩展建议

## 2. 初始化目标

本次前端初始化不是只生成一个 Vite 默认模板，而是直接搭建一套可继续演进的最小前端底座。目标如下：

1. 能独立运行本地开发环境，并与后端 `/api` 成功联调。
2. 具备最基础的页面骨架、路由系统和视觉方向。
3. 具备真实鉴权链路，而不是写死 mock。
4. 为后续业务模块保留清晰的目录边界。
5. 让后续新增模块时尽量不推翻当前结构。

## 4. 技术选型落地

### 4.1 基础框架

当前初始化实际落地如下：

- 语言：TypeScript 5.7
- UI 框架：React 19
- 路由：React Router 7
- 构建工具：Vite 6
- 样式：Tailwind CSS 4
- 包管理：pnpm

### 4.2 选型理由

#### React 19

用于承接后续中后台型页面、动态表单、状态驱动 UI 与逐步增强的 AI 交互能力。当前项目体量不大，但未来会涉及较多页面状态和业务流，React 生态足够稳妥。

#### TypeScript 5.7

前端会直接对接后端真实接口，TypeScript 可以把请求体、响应体、鉴权状态、页面局部状态全部显式类型化，减少联调时的低级错误。

#### Vite 6

本地启动快，适合作为 DailyForge 这种迭代频繁的前端工程基础脚手架。后续如果继续保留单体 Web 前端，Vite 足够支撑。

#### Tailwind CSS 4

当前阶段页面还在快速试探布局与视觉方向，Tailwind 能让页面开发速度更高，也便于后面在没有设计系统之前先快速收敛视觉语言。

#### pnpm

安装快，依赖隔离更好，适合本项目未来前后端分目录并行开发的方式。

## 6. 路由设计

当前路由定义集中在 `src/app/router.tsx`，原因是项目还处于初始化阶段，路由数量不多，集中定义更直观。

当前已定义的路由：

| 路径 | 页面 | 是否需要登录 |
|------|------|------|
| `/` | LandingPage | 否 |
| `/login` | LoginPage | 否 |
| `/register` | RegisterPage | 否 |
| `/app` | HomePage | 是 |
| `/invite-code` | RedeemInviteCodePage | 是 |

### 6.1 路由守卫策略

项目通过 `ProtectedOutlet` 实现第一版前端路由守卫：

1. 应用启动后先检查本地是否有保存的 `accessToken`。
2. 如果有，则调用 `GET /api/auth/me` 校验登录态。
3. 校验期间显示“正在同步账号状态”加载占位。
4. 如果校验失败，则清空本地会话并跳转到登录页。
5. 如果校验成功，则允许进入受保护页面。

这种设计的好处是：

- 前端不会只相信本地 token 是否存在。
- 受保护页面拿到的是后端最新用户状态。
- 后续接角色权限和会员等级判断时可以直接复用。

## 7. 鉴权状态设计

### 7.1 当前实现位置

鉴权状态由 `src/app/providers/AuthProvider.tsx` 统一管理。

暴露给页面的能力包括：

- `currentUser`
- `isAuthenticated`
- `isBootstrapping`
- `accessToken`
- `login`
- `register`
- `logout`
- `redeemInviteCode`

### 7.2 状态流转

#### 启动阶段

1. 从 `localStorage` 读取会话。
2. 如果没有会话，直接结束 bootstrap。
3. 如果有会话，则调用 `/api/auth/me`。
4. 成功后写入 `currentUser`。
5. 失败后清空会话并回到未登录态。

#### 登录阶段

1. 页面提交邮箱和密码。
2. 调用 `/api/auth/login`。
3. 后端返回 `accessToken + refreshToken + user`。
4. 前端把 token 与用户摘要写入 `localStorage`。
5. 同时把 `currentUser` 写入内存状态。

#### 退出阶段

1. 调用 `/api/auth/logout`。
2. 无论接口是否成功，都清空本地会话。
3. 重置 `currentUser` 和 `session`。

#### 邀请码兑换阶段

1. 调用 `/api/auth/redeem-invite-code`。
2. 接口成功后，只更新 `currentUser.accountTier`。
3. 当前版本不会重新请求 `/api/auth/me`。

### 7.3 当前边界

当前实现是最小可用版本，尚未覆盖：

- 自动刷新 access token
- 401 自动续签
- 多标签页同步登录态
- 登录过期前提示
- 更细的角色权限守卫

## 8. 接口请求设计

### 8.1 通用封装

`src/shared/api/http.ts` 提供了一个轻量的通用 `request<T>` 方法。

当前职责：

- 自动拼接 `/api`
- 自动设置 `Content-Type: application/json`
- 如果传入 `accessToken`，自动追加 `Authorization: Bearer xxx`
- 统一解析后端 `ApiResponse<T>`
- 对非 2xx 响应抛出 `Error`

### 8.2 当前优点

- 简单直接，适合项目初始阶段
- 与后端统一响应模型完全对齐
- 业务模块不需要重复写 fetch 基础逻辑

### 8.3 当前缺点

当前也存在几个明显边界：

1. 错误只保留 `message`，没有保留 `code`。
2. 没有统一超时控制。
3. 没有拦截器机制。
4. 没有自动刷新 token。
5. 没有区分网络错误和业务错误。

后续如果模块继续增多，建议把当前 `request` 升级为更完整的 API 客户端层。

## 9. 页面设计概览

### 9.1 LandingPage

作用：

- 作为未登录首页
- 说明项目定位
- 展示已接通的前端基础能力
- 根据登录状态引导到注册页或控制台

### 9.2 LoginPage

作用：

- 提交登录信息
- 成功后跳转到 `/app`
- 失败后展示错误信息

### 9.3 RegisterPage

作用：

- 提交注册信息
- 前端先校验两次密码是否一致
- 成功后跳转到 `/login`

### 9.4 RedeemInviteCodePage

作用：

- 已登录用户兑换邀请码
- 验证当前 `auth` 模块里“用户升级账户层级”的接口链路

### 9.5 HomePage

作用：

- 作为当前登录后的临时控制台首页
- 展示当前用户基本信息
- 为后续业务模块接入提供导航落点

## 10. 视觉与样式设计

### 10.1 当前视觉方向

当前页面没有走“默认白底后台模板”路线，而是先建立一套更贴合健身主题的基础视觉语言：

- 深色暖棕背景
- 琥珀色高亮
- 半透明卡片
- 背景网格与径向光效
- 较大的圆角和较强的层次阴影

这样做的原因是希望 DailyForge 从第一版开始就有明确气质，不至于后面再整站推翻。

### 10.2 样式组织方式

当前采用：

- Tailwind 原子类驱动页面样式
- `src/styles/index.css` 存放全局基础样式

`index.css` 当前负责：

- 引入 Tailwind
- 定义字体栈
- 设置根节点背景与文本渲染
- 全局 `box-sizing`
- 重置基础元素样式

### 10.3 当前边界

当前还没有：

- 设计 token 文件
- 主题变量分层
- 通用组件样式抽象
- 响应式规范文档
- 动效规范文档

后续当页面数量增加后，建议把颜色、圆角、间距和阴影提取成更明确的视觉 token。

## 11. 本地开发与运行设计

### 11.1 入口命令

前端当前支持：

```bash
pnpm dev
pnpm build
pnpm preview
```

### 11.2 本地联调方式

Vite 通过 `vite.config.ts` 将前端 `/api` 请求代理到：

`http://localhost:8080`

因此前端代码统一写：

```ts
fetch("/api/auth/login")
```

而不是写死完整后端地址。

这样做的好处是：

- 开发态避免跨域
- 以后切环境时更容易切换
- 前端代码不用散落后端域名

### 11.3 pnpm 工作区补充说明

当前 `frontend/pnpm-workspace.yaml` 已配置：

- `packages: ["."]`
- `allowBuilds.esbuild: true`

这是为了解决：

- pnpm 识别当前目录为工作区根目录时必须声明 `packages`
- esbuild 在 pnpm 11 下需要显式允许 postinstall 构建脚本

## 12. 初始化过程中遇到的问题与处理

### 12.1 pnpm workspace 配置问题

现象：

- `pnpm dev` 报错 `packages field missing or empty`

原因：

- `pnpm-workspace.yaml` 只有 `allowBuilds`，没有 `packages`

处理：

- 增加 `packages: ["."]`

### 12.2 esbuild 构建脚本审批问题

现象：

- pnpm 安装完成后提示 `Ignored build scripts: esbuild`

原因：

- pnpm 11 默认对构建脚本进行审批

处理：

- 在 `pnpm-workspace.yaml` 中显式配置 `allowBuilds.esbuild: true`

### 12.3 沙箱环境下的 `spawn EPERM`

现象：

- 在当前受限执行环境里，`vite` / `esbuild` 启动时可能报 `spawn EPERM`

原因：

- 不是项目代码错误，而是当前执行环境对子进程启动的限制

结论：

- 在正常本地环境中前端可以启动并构建成功

## 13. 当前已知问题

### 13.1 源码中的部分中文字符串编码异常

当前部分源码文件中的中文文本已经出现乱码字符，这不影响文档设计本身，但会影响：

- 页面文案显示
- 后续代码阅读体验
- 文档与实现的一致性

建议后续尽快统一前端源码文件编码为 UTF-8，并顺手把当前页面文案清洗一遍。

### 13.2 `AuthProvider` 对登录响应的类型假设偏乐观

`login` 成功后直接从 `response.user.status` 取值，而后端 `AuthTokenResponse.user` 是否长期包含 `status` 需要和后端文档保持一致。后续如果后端摘要对象调整，需要同步修正前端类型。

### 13.3 当前控制台仍然是占位页

`/app` 目前只是联调落点，不是正式业务首页。
