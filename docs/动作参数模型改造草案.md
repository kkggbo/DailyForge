# DailyForge 动作参数模型改造草案

> 版本：v0.1  
> 日期：2026-07-15  
> 适用范围：`cycle_template` 模块下一阶段改造评审  
> 目标读者：前端 agent、后端 agent、产品需求确认

---

## 1. 背景

当前 `cycle_template` 模块中的单个动作，采用固定字段模型描述目标参数，主要字段包括：

- `targetSets`
- `targetRepsMin`
- `targetRepsMax`
- `targetWeightKg`
- `targetDurationSeconds`
- `restSeconds`
- `targetRpe`
- `note`
- `targetExtraJson`

这套模型的优点是：
- 实现简单
- 表单直接
- 接口和数据库结构清晰

但问题也很明显：
- 不同动作需要的参数差异很大
- 同一个动作的不同组，参数可能不同
- 有氧动作不一定需要“组数”
- `targetExtraJson` 虽然能扩展，但语义不稳定，不利于前后端统一和后续统计/AI 分析

因此，需要评估是否将“固定字段动作模型”升级为“结构化动作参数模型”。

---

## 2. 本次改造要解决的问题

### 2.1 当前模型不足

例如以下场景很难被现有模型自然表达：

1. 卧推 4 组，每组重量和次数不同
2. 深蹲前两组热身，后两组正式组
3. 跑步只需要时长、速度、坡度，不需要组数
4. 间歇跑包含多个阶段，每个阶段参数不同
5. 后续训练打卡时，需要按组/按阶段记录“计划值”和“实际值”

### 2.2 改造目标

本次改造希望支持：

1. 不同动作可使用不同参数结构
2. 同一动作下可存在多个执行单元
3. 每个执行单元可拥有不同参数
4. 训练模板与未来训练打卡使用同一套结构语义
5. 为后续统计与 AI 分析保留结构化数据基础

---

## 3. 不建议直接做“完全自由参数拼装”

从产品能力上看，“用户自由添加任意参数”是可行的，但不建议直接一步做成完全自由模型。

原因：

1. 用户理解成本高  
普通用户不容易理解“一级参数 / 二级参数”的结构编辑方式。

2. 前端复杂度显著上升  
这会让模板编辑页从表单变成一个小型结构编辑器，涉及动态增删、排序、局部校验、撤销、重置、复制等复杂交互。

3. 后端模型会大改  
当前 DTO、VO、Entity、DDL 都是围绕固定字段设计，完全自由化会触发中等规模重构。

4. 会联动影响 `training session`  
模板能这样配置，训练打卡就也必须支持同结构记录计划和实际完成情况。

因此，本草案建议走：

**“半结构化模型”而不是“完全自由模型”**

即：
- 系统先按动作类型给出推荐结构
- 用户在允许范围内增删执行项和参数
- 不从纯空白开始自由拼装

---

## 4. 推荐的核心抽象

不建议继续使用“一级参数 / 二级参数”作为产品概念。  
建议统一为以下三层抽象：

1. 动作
2. 执行项
3. 执行项参数

---

## 5. 概念定义

### 5.1 动作

动作是模板中的一个训练动作，例如：
- 杠铃卧推
- 深蹲
- 跑步
- 椭圆机

动作负责表达：
- 选中的系统动作
- 在当天的排序
- 动作的结构类型

### 5.2 执行项

执行项是一个动作下的可执行单元。

不同动作的执行项语义不同：

- 力量训练动作：通常是“第 1 组 / 第 2 组 / 第 3 组”
- 持续有氧动作：通常是“主训练段”
- 间歇有氧动作：通常是“第 1 段 / 第 2 段 / 第 3 段”

### 5.3 执行项参数

执行项参数是该执行单元上的具体参数，例如：

- 重量
- 次数
- 时长
- 距离
- 速度
- 配速
- 坡度
- RPE
- 休息
- 备注

---

## 6. 推荐的结构类型

MVP 不建议一开始支持无限结构类型。  
建议先收敛为以下两类主结构。

### 6.1 `set_based` 按组动作

适用场景：
- 力量训练
- 器械训练
- 自重训练

特点：
- 一个动作下可配置多个“组”
- 每组参数可不同

常见参数：
- `weight_kg`
- `reps`
- `duration_seconds`
- `rpe`
- `rest_seconds`
- `note`

示例：

杠铃卧推：
- 第 1 组：40kg x 12
- 第 2 组：50kg x 10
- 第 3 组：60kg x 8
- 第 4 组：60kg x 8

### 6.2 `single_segment` 单段动作

适用场景：
- 跑步
- 爬坡
- 椭圆机
- 骑行

特点：
- 一个动作默认只有 1 个执行项
- 不强调组数

常见参数：
- `duration_seconds`
- `distance_km`
- `speed_kmh`
- `pace_seconds_per_km`
- `incline_percent`
- `intensity_level`
- `note`

示例：

跑步：
- 主训练段：20 分钟 / 速度 6.5 / 坡度 6

---

## 7. 后续可扩展结构类型

以下类型建议不在首轮改造中实现，但在模型设计时预留：

- `interval_segment`：间歇段
- `warmup_working_set`：热身组 / 正式组区分
- `dropset`：递减组
- `superset`：超级组
- `circuit`：循环训练

---

## 8. 推荐的产品交互方案

### 8.1 不推荐的交互

不推荐：
- 用户选中动作后，从完全空白状态自己自由添加所有参数结构

原因：
- 太像“配置编辑器”
- 不符合大多数用户直觉

### 8.2 推荐的交互

推荐：

1. 用户先选择系统动作
2. 系统根据动作类型带出默认结构
3. 用户在默认结构上增删执行项
4. 用户在执行项中增删允许的参数

例如：

#### 卧推

选择“杠铃卧推”后，默认结构：
- 按组动作
- 自动生成 1 组空白执行项

用户可：
- 增加到 4 组
- 每组填写不同重量、次数、RPE

#### 跑步

选择“跑步”后，默认结构：
- 单段动作
- 自动生成 1 个主训练段

用户可：
- 添加时长
- 添加速度
- 添加坡度
- 不显示组数

### 8.3 草稿编辑页推荐体验

对于模板编辑页，建议：

1. 动作层级显示为卡片
2. 动作卡片内展示执行项列表
3. 每个执行项显示可选参数区
4. 参数显示顺序由系统定义，不建议完全自由拖拽字段

这样可以兼顾灵活性和可理解性。

---

## 9. 推荐的数据模型

## 9.1 当前模型

当前基本等价于：

### 动作表

`cycle_day_exercises`

字段里直接包含：
- 组数
- 次数范围
- 重量
- 时长
- 休息
- RPE
- 备注
- extra json

这个模型更像：

**“动作 + 一整套固定目标字段”**

---

## 9.2 推荐模型

建议重构为三层：

### 第一层：模板动作表

建议保留 `cycle_day_exercises` 作为动作主表，但弱化固定参数字段。

建议新增或保留的核心字段：
- `id`
- `template_day_id`
- `exercise_id`
- `exercise_name_snapshot`
- `sort_order`
- `structure_type`
- `note`

建议逐步废弃的固定字段：
- `target_sets`
- `target_reps_min`
- `target_reps_max`
- `target_weight_kg`
- `target_duration_seconds`
- `rest_seconds`
- `target_rpe`
- `target_extra_json`

### 第二层：模板动作执行项表

建议新增表：

`cycle_day_exercise_items`

建议字段：
- `id`
- `cycle_day_exercise_id`
- `item_index`
- `item_type`
- `item_name`
- `sort_order`
- `note`
- `created_at`
- `updated_at`

说明：
- `item_type` 取值可为：`set`、`segment`
- `item_name` 可选，例如“第1组”“主训练段”

### 第三层：模板动作执行项参数表

建议新增表：

`cycle_day_exercise_item_metrics`

建议字段：
- `id`
- `exercise_item_id`
- `metric_key`
- `metric_value_text`
- `metric_value_number`
- `metric_unit`
- `sort_order`
- `created_at`
- `updated_at`

说明：
- `metric_key` 例如：`weight_kg`、`reps`、`duration_seconds`、`speed_kmh`
- `metric_value_number` 存大多数可计算参数
- `metric_value_text` 用于备注类字段
- `metric_unit` 例如：`kg`、`seconds`、`km/h`

---

## 10. 为什么推荐“三层表模型”而不是只堆 JSON

### 10.1 纯 JSON 方案

可以把整个动作结构都存到一个 JSON 字段里，例如：
- `structure_json`

优点：
- 开发快
- 表结构改动少

缺点：
- 难校验
- 难查询
- 难做统计
- 难做 SQL 层筛选与演进

### 10.2 三层结构化方案

优点：
- 参数语义清晰
- 更适合未来训练打卡复用
- 更适合统计分析
- 更适合 AI 读取结构化计划内容

缺点：
- 实现复杂度更高

### 10.3 推荐结论

如果只追求短期改动最小，可以做 JSON。  
如果希望这套模型后续复用到 `training session` 和统计，推荐直接走三层结构化方案。

本草案的推荐结论是：

**优先采用三层结构化方案。**

## 11. 前后端接口建议

## 11.1 当前接口问题

当前请求/响应围绕固定字段，例如：
- `targetSets`
- `targetRepsMin`
- `targetRepsMax`

改造后，这种 DTO 需要整体升级。

## 11.2 推荐的请求结构

建议未来请求结构改为：

```json
{
  "exerciseId": 1001,
  "structureType": "set_based",
  "note": "主动作",
  "items": [
    {
      "itemIndex": 1,
      "itemType": "set",
      "itemName": "第1组",
      "metrics": [
        { "metricKey": "weight_kg", "metricValueNumber": 40, "metricUnit": "kg" },
        { "metricKey": "reps", "metricValueNumber": 12, "metricUnit": "count" },
        { "metricKey": "rpe", "metricValueNumber": 7.5, "metricUnit": null }
      ]
    },
    {
      "itemIndex": 2,
      "itemType": "set",
      "itemName": "第2组",
      "metrics": [
        { "metricKey": "weight_kg", "metricValueNumber": 50, "metricUnit": "kg" },
        { "metricKey": "reps", "metricValueNumber": 10, "metricUnit": "count" }
      ]
    }
  ]
}
```

## 11.3 推荐的响应结构

响应建议与请求保持同构，避免前端再做复杂结构转换。

## 12. 前端改造范围

如果采用本草案，前端主要受影响的部分包括：

### 12.1 类型定义

当前：
- `CycleTemplateExerciseResponse`
- `EditorExerciseForm`

需要升级为：
- 动作
- 执行项
- 执行项参数

### 12.2 编辑器组件

当前编辑器是固定字段表单。  
改造后将变为：
- 动作卡片
- 执行项列表
- 参数编辑区

### 12.3 校验逻辑

当前校验是固定字段校验。  
改造后需要变为：
- 结构类型校验
- 执行项最小数量校验
- 参数合法性校验
- 不同参数类型的值范围校验

### 12.4 本地编辑体验

需要继续保留：
- 本地未保存编辑
- 撤回上一步操作
- 重置为上次已保存内容

但撤回粒度会从“普通表单字段”变成“结构编辑”。

## 13. 后端改造范围

### 13.1 DTO / VO

需要重构：
- `CycleTemplateExerciseRequest`
- `CycleTemplateExerciseResponse`

### 13.2 应用服务

需要改造：
- 创建草稿
- 更新草稿
- 更新正式模板
- 详情查询组装逻辑

### 13.3 持久化层

需要新增：
- 执行项 Entity / Mapper
- 执行项参数 Entity / Mapper

### 13.4 领域规则

需要增加：
- 不同结构类型允许哪些参数
- 不同参数值范围规则
- 某些参数是否允许重复
- 单段动作是否允许多个执行项

## 14. 对 `training session` 的影响

这是本次改造最需要提前意识到的一点：

**动作参数模型一旦升级，训练打卡模型最好同步采用同一结构。**

原因：
- 模板里已经是“动作 -> 执行项 -> 参数”
- 打卡时也应该按“动作 -> 执行项 -> 参数”记录计划和实际

例如：

模板计划：
- 第1组 60kg x 8

打卡实际：
- 第1组 55kg x 6
- 原因：热身过量导致三头提前力竭

如果模板是结构化，打卡却还是固定字段，会导致后续映射很痛苦。

因此建议：

`cycle_template` 的这次改造，不要孤立看待，要视为未来 `training session` 数据模型的前置设计。

---

## 15. 推荐的 MVP 改造范围

为了控制复杂度，本草案建议首轮只做到：

### 15.1 支持两种结构类型

- `set_based`
- `single_segment`

### 15.2 支持有限参数集合

建议首批支持：
- `weight_kg`
- `reps`
- `duration_seconds`
- `distance_km`
- `speed_kmh`
- `pace_seconds_per_km`
- `incline_percent`
- `rest_seconds`
- `rpe`
- `intensity_level`
- `note`

### 15.3 不做的内容

首轮不做：
- 完全自由自定义参数 key
- 超级组
- 掉重组
- 间歇训练多段模板
- 热身组/正式组特殊语义
- 参数模板市场化复用

## 17. 推荐结论

### 17.1 总结判断

用户按动作类型自定义参数结构，这个方向是合理的，而且能更真实地表达训练计划。

### 17.2 不建议的实现方式

不建议直接做成：
- 用户完全自由拼装任意参数结构

### 17.3 推荐的实现方式

推荐做成：

- 基于动作类型的半结构化模型
- 使用“动作 -> 执行项 -> 执行项参数”三层抽象
- 首轮只支持 `set_based` 和 `single_segment`
- 数据层优先采用结构化三层表，而不是单纯依赖 JSON

### 17.4 对项目节奏的建议

这项改造是值得做的，但它本质上已经不是小修小补，而是一次会影响模板、打卡、统计和 AI 的数据模型升级。  
因此建议前后端 agent 在真正动手前，先基于本草案确认：

1. 是否接受改造范围扩大
2. 是否先做收敛版 MVP
3. 是否同步考虑 `training session` 的数据模型
