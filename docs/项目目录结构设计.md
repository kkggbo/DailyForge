# DailyForge 项目目录结构设计

## 1. 设计目标

DailyForge 的项目目录结构设计遵循以下原则：

- 根目录命名保持朴素清晰
- 仓库只服务 Web 项目，不为多端场景做额外抽象
- 前后端分离
- 后端采用模块化单体架构
- 前后端都优先按业务边界组织目录
- 为 AI 能力预留明确位置，但不单独拆成独立服务

## 2. 推荐根目录结构

```text
DailyForge/
├─ frontend/
├─ backend/
├─ docs/
└─ deploy/
```

各目录职责如下：

- `frontend`：前端项目代码
- `backend`：后端项目代码
- `docs`：产品、技术、架构、接口、数据库等文档
- `deploy`：容器化部署、Nginx、环境变量模板等部署相关配置

## 3. 整体设计思路

本项目采用单仓库管理，但不做过重的 monorepo 工程化封装。  
原因是当前阶段以个人开发和 MVP 交付为主，重点是降低维护成本和联调复杂度，而不是为了未来可能的多端、多服务场景提前堆叠抽象。

目录设计的核心原则是：

- 根目录保持简洁
- 业务代码不要散落
- 通用基础设施与业务模块分开
- 模块边界尽量与未来服务边界一致

## 4. 前端目录设计

### 4.1 前端设计原则

前端目录采用“基础设施 + 业务模块”结构，而不是单纯按 `components`、`pages`、`utils` 做全局平铺。

这样设计的原因是：

- 后续维护时更容易按业务定位代码
- 每个功能模块的页面、组件、请求、类型可以尽量内聚
- 避免项目变大后通用目录无限膨胀

### 4.2 推荐前端目录结构

```text
frontend/
├─ public/
├─ src/
│  ├─ app/
│  ├─ components/
│  ├─ features/
│  │  ├─ auth/
│  │  ├─ profile/
│  │  ├─ plan/
│  │  ├─ workout/
│  │  ├─ nutrition/
│  │  ├─ stats/
│  │  └─ ai/
│  ├─ services/
│  ├─ lib/
│  ├─ types/
│  └─ styles/
├─ index.html
├─ package.json
├─ pnpm-lock.yaml
├─ tsconfig.json
└─ vite.config.ts
```

### 4.3 前端各目录职责

#### `src/app`

用于放置应用级基础设施，例如：

- 应用入口
- 路由注册
- 全局 Provider
- 全局布局
- 鉴权守卫

#### `src/components`

用于放置真正跨模块复用的通用组件，例如：

- 按钮
- 弹窗
- 表格
- 表单基础组件
- 通用图表容器

这里不放强业务含义的组件，避免“看起来通用，实际上只服务一个模块”的情况。

#### `src/features`

用于放置按业务划分的功能模块，是前端最核心的目录。

建议初始模块如下：

- `auth`：注册、登录、鉴权状态
- `profile`：用户基础资料、目标设置、身体指标录入
- `plan`：训练计划生成、查看、编辑
- `workout`：训练打卡、训练详情记录
- `nutrition`：饮食建议、热量和营养素展示
- `stats`：历史统计、趋势图表、周期总结
- `ai`：AI 结果展示、AI 请求输入、AI 建议交互

每个模块内部可以按需要继续细分，例如：

```text
plan/
├─ api/
├─ components/
├─ hooks/
├─ pages/
├─ schemas/
└─ types/
```

说明：

- `pages` 放该模块页面
- `components` 放该模块专属组件
- `api` 放该模块接口调用封装
- `schemas` 放 Zod 校验结构
- `types` 放该模块本地类型定义

#### `src/services`

用于放置全局请求层和跨模块服务能力，例如：

- HTTP client
- 请求拦截器
- Token 注入逻辑
- 通用 API 响应处理

#### `src/lib`

用于放置通用工具和基础能力，例如：

- 日期处理
- 数值格式化
- 常量
- 工具函数

#### `src/types`

用于放置全局共享类型，例如：

- 通用分页结构
- 统一响应结构
- 全局用户上下文类型

#### `src/styles`

用于放置全局样式入口和主题样式，例如：

- Tailwind 全局入口
- 自定义 CSS 变量
- 字体与基础样式

### 4.4 前端设计约束

- 页面优先放在所属业务模块下，不集中塞入一个全局 `pages`
- 业务组件优先放所属模块，不要过早抽到全局 `components`
- AI 相关前端逻辑只负责交互和结果展示，不承载核心业务规则

## 5. 后端目录设计

### 5.1 后端设计原则

后端采用模块化单体架构，不在 MVP 阶段拆微服务。

但代码组织上提前体现业务边界，做到：

- 对内是单体，便于开发和部署
- 对代码结构来说是模块化的，便于后续演进

不推荐采用全局平铺式结构，例如：

- 所有 `controller` 放一个目录
- 所有 `service` 放一个目录
- 所有 `mapper` 放一个目录

这种方式在项目初期看起来简单，但后续会导致业务边界混乱、查找路径拉长、拆分困难。

### 5.2 推荐后端目录结构

```text
backend/
├─ src/
│  ├─ main/
│  │  ├─ java/com/dailyforge/
│  │  │  ├─ common/
│  │  │  ├─ config/
│  │  │  ├─ infrastructure/
│  │  │  ├─ modules/
│  │  │  │  ├─ auth/
│  │  │  │  ├─ user/
│  │  │  │  ├─ profile/
│  │  │  │  ├─ plan/
│  │  │  │  ├─ workout/
│  │  │  │  ├─ nutrition/
│  │  │  │  ├─ stats/
│  │  │  │  └─ ai/
│  │  │  └─ DailyForgeApplication.java
│  │  └─ resources/
│  │     ├─ mapper/
│  │     ├─ db/migration/
│  │     ├─ application.yml
│  │     ├─ application-dev.yml
│  │     └─ application-prod.yml
│  └─ test/
├─ pom.xml
└─ README.md
```

### 5.3 后端各目录职责

#### `common`

用于放置全局共享但不属于某个业务模块的能力，例如：

- 统一返回结构
- 通用异常体系
- 错误码
- 常量
- 通用工具类

#### `config`

用于放置 Spring 配置类，例如：

- Web 配置
- 跨域配置
- Jackson 配置
- OpenAPI 配置

#### `infrastructure`

用于放置基础设施实现，例如：

- JWT 能力
- Redis 配置
- MyBatis-Plus 配置
- 安全框架接入
- 外部 HTTP 客户端配置
- Trace / 日志上下文接入

说明：

这里放的是“技术基础设施”，不是业务逻辑。

#### `modules`

用于放置核心业务模块，是后端主干目录。

建议初始模块如下：

- `auth`：注册、登录、Token 刷新、认证上下文
- `user`：账号主体信息
- `profile`：身体信息、训练目标、训练基础信息
- `plan`：训练计划生成、计划版本、计划编辑
- `workout`：训练记录、动作记录、组数次数重量
- `nutrition`：热量目标、营养素建议、饮食策略
- `stats`：历史统计、趋势分析、周期总结
- `ai`：模型调用、Prompt 模板、响应解析、调用日志

### 5.4 模块内推荐结构

每个业务模块内部建议采用统一结构，例如：

```text
plan/
├─ controller/
├─ service/
├─ mapper/
├─ entity/
├─ dto/
├─ vo/
└─ convert/
```

各子目录职责如下：

- `controller`：提供 HTTP 接口
- `service`：承载业务逻辑
- `mapper`：数据库访问层
- `entity`：数据库实体对象
- `dto`：请求参数和内部传输对象
- `vo`：返回给前端的视图对象
- `convert`：对象转换逻辑，例如 MapStruct

如果模块后续复杂度增加，还可以补充：

- `enums`
- `repository`
- `domain`
- `event`

但 MVP 阶段不建议提前过度细分。

## 6. AI 模块设计原则

### 6.1 为什么要单独预留 `ai` 模块

DailyForge 计划较早接入 AI，因此应在结构上提前预留明确位置，避免后续把模型调用逻辑散落到 `plan`、`nutrition`、`stats` 等业务模块中。

### 6.2 `ai` 模块职责

`ai` 模块建议负责以下能力：

- Prompt 模板管理
- 模型调用封装
- 请求参数组装
- 响应结果解析
- 调用日志记录
- 超时、重试、限流、降级

### 6.3 `ai` 模块不负责什么

`ai` 模块不应该直接承载以下决策：

- 训练计划的核心业务规则
- 饮食目标的核心计算规则
- 统计口径定义

这些规则仍应分别属于：

- `plan`
- `nutrition`
- `stats`

换句话说：

- 业务模块决定“什么时候调用 AI”
- `ai` 模块负责“如何调用 AI”

## 7. 数据与配置目录设计

### 7.1 数据库迁移

数据库结构变更建议统一放在：

```text
backend/src/main/resources/db/migration/
```

推荐使用 Flyway 管理数据库版本，而不是长期依赖手写零散 SQL 文件。

这样做的好处是：

- 表结构演进可追踪
- 开发环境和测试环境更容易同步
- 后续部署时更容易自动化

### 7.2 Mapper XML

如果使用 MyBatis-Plus 且存在自定义 SQL，建议 XML 放在：

```text
backend/src/main/resources/mapper/
```

并按业务模块进一步划分子目录，例如：

```text
mapper/
├─ plan/
├─ workout/
└─ stats/
```

### 7.3 环境配置

基础配置建议放在：

- `application.yml`
- `application-dev.yml`
- `application-prod.yml`

这样可以满足本地开发和部署环境切换需求。  
如果未来确实引入 Nacos Config，再将外部配置中心作为第二阶段扩展。

## 8. 部署目录设计

推荐部署目录结构如下：

```text
deploy/
├─ docker-compose.yml
├─ nginx/
└─ env/
```

各目录职责如下：

- `docker-compose.yml`：本地和单机部署编排
- `nginx`：反向代理配置、静态资源转发配置
- `env`：环境变量模板

后续可按需补充：

- `mysql/`
- `redis/`
- `scripts/`

但不建议一开始堆太多与 MVP 无关的部署脚本。

## 9. 文档目录设计建议

`docs` 目录建议逐步维护以下内容：

- `MVP需求清单 v1.md`
- `技术选型方案.md`
- `项目目录结构设计.md`
- `数据库设计.md`
- `API设计.md`
- `开发计划.md`

这样可以保证产品、技术和实现路径都在仓库内可追踪。

## 11. 结论

本方案适合当前的 DailyForge 阶段，原因如下：

- 命名简单直接
- 结构清晰，不为多端或微服务过早设计
- 前后端都以业务边界为中心
- AI 能力有独立位置，但不破坏主体架构
- 未来如果业务增长，仍有足够演进空间

这套目录结构的核心不是“看起来复杂”，而是让未来每个功能都能找到明确归属，避免项目在增长过程中逐步失控。
