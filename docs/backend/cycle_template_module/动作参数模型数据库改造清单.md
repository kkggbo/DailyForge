# DailyForge 动作参数模型数据库改造清单

> 版本：v1.1  
> 日期：2026-07-15  
> 适用范围：`cycle_template` 模块动作参数模型重构  
> 前置结论：采用三层结构化模型，接受破坏式迁移

---

## 1. 文档目标

本文档用于把“动作参数模型改造”落成数据库层面的正式改造清单，作为后续：

- Flyway 迁移脚本设计
- `cycle_template` 接口文档 v2
- `cycle_template` DDD v2
- Java Entity / Mapper / DTO / VO 重构

的直接输入。

---

## 2. 已确认的设计决策

本次数据库改造基于以下已确认决策：

1. 不再支持 `8~10 次` 这种范围型目标。
2. 动作默认结构类型通过 `exercises` 显式字段提供。
3. `single_segment` 在 MVP 只允许 1 个执行项。
4. 接受破坏式迁移，不保留旧模型兼容层。
5. `note` 保留在动作层和执行项层，不作为 metric。
6. metric 字典在 MVP 为封闭集合，不支持用户自定义 key。
7. 同一个执行项下同一个 `metric_key` 不允许重复。
8. `structure_type` 由动作元数据决定，MVP 不支持用户随意切换。
9. `single_segment` 不按动作强制某个固定必填 metric，只要求至少存在 1 个 metric。
10. `metricUnit` 不由前端提交，不在数据库中存储，由后端按 `metric_key` 推导。
11. 当前 MVP 的所有 `metric_key` 都视为数值型参数，因此不保留 `metric_value_text`。
12. 迁移前直接清空全部 `cycle_template` 相关业务数据，不做旧模板结构迁移。

---

## 3. 改造目标

当前模型：

- `cycle_day_exercises` 一张表同时承担：
  - 动作头
  - 动作参数
  - 组信息近似表达

目标模型：

- 动作头：`cycle_day_exercises`
- 执行项：`cycle_day_exercise_items`
- 执行项参数：`cycle_day_exercise_item_metrics`

新的结构化关系为：

```text
cycle_template_days
  -> cycle_day_exercises
      -> cycle_day_exercise_items
          -> cycle_day_exercise_item_metrics
```

---

## 4. 总体改造策略

由于本次接受破坏式迁移，数据库策略采用“一刀切重构”：

1. 直接移除 `cycle_day_exercises` 上旧固定参数列。
2. 新增动作默认结构字段到 `exercises`。
3. 新增执行项表与执行项参数表。
4. 不做旧模板数据迁移兼容。
5. 迁移执行前清空全部 `cycle_template` 相关业务数据。
6. 后续代码、测试、接口文档统一切到新结构。

这意味着迁移完成后：

- 旧版 `cycle_template` 请求体会失效
- 旧版模板详情响应结构会失效
- 旧版 Java Entity / DTO / VO / 测试数据必须同步更新

---

## 5. 迁移前数据处理

## 5.1 处理原则

本次不对旧模板数据做结构迁移。

原因：

- 旧模型是固定字段
- 新模型是三层结构
- 当前已确认接受破坏式迁移

因此迁移执行前必须先清空全部 `cycle_template` 相关业务数据，避免留下缺少 item / metric 结构的不可用模板。

## 5.2 需要清空的表

建议清空顺序如下：

1. `user_active_cycles`
2. `training_session_sets`
3. `training_session_exercises`
4. `training_sessions`
5. `cycle_runs`
6. `cycle_day_exercise_item_metrics`
7. `cycle_day_exercise_items`
8. `cycle_day_exercises`
9. `cycle_template_days`
10. `cycle_template_versions`
11. `cycle_templates`

说明：

- 其中 `cycle_day_exercise_item_metrics`、`cycle_day_exercise_items` 是新表，首次迁移时可能还不存在
- 真正写 Flyway 时，可把数据清空步骤放到独立维护脚本或迁移前人工清理步骤中

## 6. 表级改造清单

## 6.1 `exercises` 表改造

### 改造目标

让动作元数据显式提供默认结构类型，支持前后端在选动作时自动带出默认结构。

### 新增字段

- `default_structure_type VARCHAR(32) NOT NULL`
  - 含义：动作默认结构类型
  - 建议值：
    - `set_based`
    - `single_segment`

### 是否新增索引

MVP 可不单独加索引。

原因：

- 该字段更多用于单条动作读取，不是高频筛选条件
- 当前动作列表主要还是按已有字段筛选

### 迁移注意事项

- 必须为现有系统动作补默认值
- 不允许为空，因为模板创建时要依赖它带出结构

### 改造后字段补充说明

现有 `ExerciseReadMapper` 后续至少需要读取：

- `id`
- `owner_user_id`
- `name`
- `is_active`
- `exercise_type`
- `movement_type`
- `default_unit`
- `default_structure_type`

同时系统动作搜索/详情接口至少需要返回：

- `exerciseId`
- `exerciseName`
- `defaultStructureType`

---

## 6.2 `cycle_day_exercises` 表改造

### 改造目标

把它从“动作 + 固定参数表”改造成“动作头表”。

### 保留字段

- `id`
- `template_day_id`
- `exercise_id`
- `exercise_name_snapshot`
- `sort_order`
- `note`

### 新增字段

- `structure_type VARCHAR(32) NOT NULL`
  - 含义：该动作实例实际采用的结构类型
  - 建议值：
    - `set_based`
    - `single_segment`

说明：

- 虽然 MVP 不允许用户切换结构类型，但模板动作实例层仍应保存 `structure_type`
- 原因是动作元数据未来可能变化，而模板历史快照不应被动作库后续改动污染

### 直接删除字段

以下字段在本次破坏式迁移中直接删除：

- `target_sets`
- `target_reps_min`
- `target_reps_max`
- `target_weight_kg`
- `target_duration_seconds`
- `target_rest_seconds`
- `target_rpe`
- `target_extra_json`

### 命名统一

本轮统一把旧列 `notes` 收敛为单数：

- `note`

### 索引调整

保留：

- `idx_cycle_day_exercises_exercise_id`
- `idx_cycle_day_exercises_day_sort (template_day_id, sort_order)`

无需新增额外索引。

### 迁移注意事项

- 删除旧字段后，所有依赖 `CycleDayExerciseEntity` 固定参数属性的代码都会编译失败
- 这是预期行为，有利于一次性收口旧模型残留

---

## 6.3 新增 `cycle_day_exercise_items` 表

### 表职责

表示一个模板动作下的可执行单元。

不同结构下语义如下：

- `set_based`
  - 每个 item 对应一组
- `single_segment`
  - 默认只允许 1 个 item，对应一个训练段

### 建议字段

- `id BIGINT UNSIGNED NOT NULL AUTO_INCREMENT`
- `cycle_day_exercise_id BIGINT UNSIGNED NOT NULL`
- `item_index SMALLINT UNSIGNED NOT NULL`
- `item_type VARCHAR(32) NOT NULL`
- `item_name VARCHAR(64) NULL`
- `note VARCHAR(500) NULL`
- `sort_order INT NOT NULL DEFAULT 0`
- `created_at DATETIME(3) NOT NULL DEFAULT CURRENT_TIMESTAMP(3)`
- `updated_at DATETIME(3) NOT NULL DEFAULT CURRENT_TIMESTAMP(3) ON UPDATE CURRENT_TIMESTAMP(3)`

### 字段说明

- `item_index`
  - 业务序号
  - 对 `set_based` 可表示第几组
  - 对 `single_segment` MVP 固定为 `1`
- `item_type`
  - 建议值：
    - `set`
    - `segment`
- `item_name`
  - 可选展示名
  - 例如：
    - `第1组`
    - `主训练段`
- `note`
  - 该执行项层级备注

### 主键与索引建议

- 主键：`PRIMARY KEY (id)`
- 唯一键：`uk_cycle_day_exercise_items_exercise_item (cycle_day_exercise_id, item_index)`
- 普通索引：`idx_cycle_day_exercise_items_exercise_sort (cycle_day_exercise_id, sort_order)`

### 外键建议

- `fk_cycle_day_exercise_items_exercise_id`
  - `cycle_day_exercise_id -> cycle_day_exercises.id`

### 约束建议

数据库层不建议硬编码：

- `set_based` 只能配 `set`
- `single_segment` 只能配 `segment`

这类规则建议放在应用层校验，避免 DDL 过度复杂。

---

## 6.4 新增 `cycle_day_exercise_item_metrics` 表

### 表职责

表示某个执行项下的结构化参数集合。

### 建议字段

- `id BIGINT UNSIGNED NOT NULL AUTO_INCREMENT`
- `exercise_item_id BIGINT UNSIGNED NOT NULL`
- `metric_key VARCHAR(64) NOT NULL`
- `metric_value_number DECIMAL(12,4) NOT NULL`
- `sort_order INT NOT NULL DEFAULT 0`
- `created_at DATETIME(3) NOT NULL DEFAULT CURRENT_TIMESTAMP(3)`
- `updated_at DATETIME(3) NOT NULL DEFAULT CURRENT_TIMESTAMP(3) ON UPDATE CURRENT_TIMESTAMP(3)`

### 字段说明

- `metric_key`
  - MVP 封闭字典，建议值：
    - `weight_kg`
    - `reps`
    - `duration_seconds`
    - `distance_km`
    - `speed_kmh`
    - `pace_seconds_per_km`
    - `incline_percent`
    - `rest_seconds`
    - `rpe`
    - `intensity_level`
- `metric_value_number`
  - 承载所有当前 MVP 支持的参数值
  - 当前这批 `metric_key` 全部视为数值型参数
  - 前端只提交 `metric_value_number`
  - 参数单位不落库，由后端按 `metric_key` 推导

### 主键与索引建议

- 主键：`PRIMARY KEY (id)`
- 唯一键：`uk_cycle_day_exercise_item_metrics_item_key (exercise_item_id, metric_key)`
- 普通索引：
  - `idx_cycle_day_exercise_item_metrics_item_sort (exercise_item_id, sort_order)`
  - 可选：`idx_cycle_day_exercise_item_metrics_key (metric_key)`

### 外键建议

- `fk_cycle_day_exercise_item_metrics_item_id`
  - `exercise_item_id -> cycle_day_exercise_items.id`

### 值存储规则

MVP 最终规则写死为：

- 所有当前支持的 `metric_key` 都是数值型
- `metric_value_number` 必填
- 不存在 `metric_value_text`
- 单位不落库，由后端推导

## 7. 是否需要新增参数字典表

本轮不建议新增参数字典表。

原因：

- 你已经确认 metric 集合为封闭集合
- 当前规模下，把字典规则维护在 Java 枚举/领域服务里更简单
- 引入表驱动字典会增加一轮后台配置复杂度，但对 MVP 收益不大

因此本轮建议：

- 数据库存实际使用的 metrics
- 参数字典和允许规则先由后端代码维护

后续如果你准备开放更多结构类型或配置后台，再考虑新增如：

- `exercise_metric_profiles`
- `exercise_metric_profile_items`

## 10. 推荐的 DDL 方向

以下不是最终可执行 SQL，只是给下一步正式 DDL 设计定边界。

### 10.1 `exercises` 新增字段

```sql
ALTER TABLE exercises
    ADD COLUMN default_structure_type VARCHAR(32) NULL COMMENT '默认动作结构类型';
```

后续再配合数据回填改为 `NOT NULL`。

### 10.2 `cycle_day_exercises` 调整

```sql
ALTER TABLE cycle_day_exercises
    ADD COLUMN structure_type VARCHAR(32) NULL COMMENT '动作结构类型',
    CHANGE COLUMN notes note VARCHAR(500) NULL COMMENT '动作备注';
```

回填后删除旧字段：

```sql
ALTER TABLE cycle_day_exercises
    DROP COLUMN target_sets,
    DROP COLUMN target_reps_min,
    DROP COLUMN target_reps_max,
    DROP COLUMN target_weight_kg,
    DROP COLUMN target_duration_seconds,
    DROP COLUMN target_rest_seconds,
    DROP COLUMN target_rpe,
    DROP COLUMN target_extra_json;
```

### 10.3 `cycle_day_exercise_items`

```sql
CREATE TABLE cycle_day_exercise_items (
    id BIGINT UNSIGNED NOT NULL AUTO_INCREMENT COMMENT '主键ID',
    cycle_day_exercise_id BIGINT UNSIGNED NOT NULL COMMENT '模板动作ID',
    item_index SMALLINT UNSIGNED NOT NULL COMMENT '执行项序号',
    item_type VARCHAR(32) NOT NULL COMMENT '执行项类型',
    item_name VARCHAR(64) NULL COMMENT '执行项名称',
    note VARCHAR(500) NULL COMMENT '执行项备注',
    sort_order INT NOT NULL DEFAULT 0 COMMENT '排序值',
    created_at DATETIME(3) NOT NULL DEFAULT CURRENT_TIMESTAMP(3) COMMENT '创建时间',
    updated_at DATETIME(3) NOT NULL DEFAULT CURRENT_TIMESTAMP(3) ON UPDATE CURRENT_TIMESTAMP(3) COMMENT '更新时间',
    PRIMARY KEY (id),
    UNIQUE KEY uk_cycle_day_exercise_items_exercise_item (cycle_day_exercise_id, item_index),
    KEY idx_cycle_day_exercise_items_exercise_sort (cycle_day_exercise_id, sort_order),
    CONSTRAINT fk_cycle_day_exercise_items_exercise_id
        FOREIGN KEY (cycle_day_exercise_id) REFERENCES cycle_day_exercises (id)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COLLATE=utf8mb4_unicode_ci COMMENT='模板动作执行项表';
```

### 10.4 `cycle_day_exercise_item_metrics`

```sql
CREATE TABLE cycle_day_exercise_item_metrics (
    id BIGINT UNSIGNED NOT NULL AUTO_INCREMENT COMMENT '主键ID',
    exercise_item_id BIGINT UNSIGNED NOT NULL COMMENT '执行项ID',
    metric_key VARCHAR(64) NOT NULL COMMENT '参数键',
    metric_value_number DECIMAL(12,4) NOT NULL COMMENT '数值参数值',
    sort_order INT NOT NULL DEFAULT 0 COMMENT '排序值',
    created_at DATETIME(3) NOT NULL DEFAULT CURRENT_TIMESTAMP(3) COMMENT '创建时间',
    updated_at DATETIME(3) NOT NULL DEFAULT CURRENT_TIMESTAMP(3) ON UPDATE CURRENT_TIMESTAMP(3) COMMENT '更新时间',
    PRIMARY KEY (id),
    UNIQUE KEY uk_cycle_day_exercise_item_metrics_item_key (exercise_item_id, metric_key),
    KEY idx_cycle_day_exercise_item_metrics_item_sort (exercise_item_id, sort_order),
    KEY idx_cycle_day_exercise_item_metrics_key (metric_key),
    CONSTRAINT fk_cycle_day_exercise_item_metrics_item_id
        FOREIGN KEY (exercise_item_id) REFERENCES cycle_day_exercise_items (id)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COLLATE=utf8mb4_unicode_ci COMMENT='模板动作执行项参数表';
```

## 12. 风险点

## 12.1 最大风险

不是建表本身，而是“旧固定字段语义”彻底退出后，所有代码都要一次切换到新结构。

## 12.2 次要风险

- `metric_key` 字典如果后续频繁变化，会引发接口和校验层反复改动
- `structure_type` 如果未来又开放手动切换，当前规则需要补充
- `single_segment` 如果后续要支持多段，会引发 item 规则放宽

## 12.3 当前可接受性

由于你已经确认：

- 接受破坏式迁移
- 不保留范围型目标
- MVP 限制 `single_segment = 1 item`

所以这些风险目前是可控的。
