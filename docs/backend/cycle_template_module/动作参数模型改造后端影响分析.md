## 1. 结论摘要

这次改造在后端上不是“小字段调整”，而是一次会穿透以下层级的中等规模重构：

- `cycle_template` 的请求/响应模型
- 模板版本快照的存储结构
- 模板保存、复制、详情查询、版本克隆逻辑
- 训练打卡的未来数据模型
- 动作元数据读取方式
- MyBatis Entity / Mapper / 测试 Schema / Flyway 迁移

基于现有代码和表结构，我的结论是：

1. 产品草案提出的“三层结构化模型”在后端是可落地的，而且比继续扩展 `targetExtraJson` 更合理。
2. 如果要做，就不建议只改 DTO，再把复杂结构塞回单个 JSON；那样会让后续训练打卡、统计、AI 分析继续背历史包袱。
3. 这次后端最合适的落地方式是：
   - 保留 `cycle_day_exercises` 作为动作头表
   - 新增“执行项表”和“执行项参数表”
   - 在 Java 侧增加结构类型、参数字典和校验规则
   - 同时把 `training_session` 的未来结构一起预对齐
4. 当前最值得提前确认的，不是表怎么拆，而是下面 4 个业务决策：
   - `8~10 次` 这种“范围型目标”要不要继续支持
   - “默认结构类型”是按动作类型推断，还是在动作表里显式配置
   - `single_segment` 在 MVP 是否只允许 1 个执行项
   - 这次是否接受破坏式迁移，还是需要兼容现有模板数据

## 2. 当前后端现状

### 2.1 当前模板动作模型是“固定字段模型”

当前 `cycle_template` 的动作参数全部直接挂在 `cycle_day_exercises` 上：

- `target_sets`
- `target_reps_min`
- `target_reps_max`
- `target_weight_kg`
- `target_duration_seconds`
- `target_rest_seconds`
- `target_rpe`
- `target_extra_json`
- `notes`

这意味着当前模型天然假设：

- 一个动作只有一套目标参数
- 组差异只能靠“统一组数 + 统一范围”表达
- 有氧类动作的特殊字段只能塞 `target_extra_json`

### 2.2 当前 Java 代码和这套固定字段强绑定

当前 `plan` 模块里，以下代码都直接依赖固定字段：

- DTO：
  - [CycleTemplateExerciseRequest.java](/D:/Computer%20Science/DailyForge/backend/src/main/java/com/dailyforge/modules/plan/interfaces/dto/CycleTemplateExerciseRequest.java)
- VO：
  - [CycleTemplateExerciseResponse.java](/D:/Computer%20Science/DailyForge/backend/src/main/java/com/dailyforge/modules/plan/interfaces/vo/CycleTemplateExerciseResponse.java)
- Entity：
  - [CycleDayExerciseEntity.java](/D:/Computer%20Science/DailyForge/backend/src/main/java/com/dailyforge/modules/plan/infrastructure/persistence/entity/CycleDayExerciseEntity.java)
- 版本快照读写：
  - [CycleTemplateVersionDomainService.java](/D:/Computer%20Science/DailyForge/backend/src/main/java/com/dailyforge/modules/plan/domain/service/CycleTemplateVersionDomainService.java)
- 详情组装：
  - [CycleTemplateQueryApplicationService.java](/D:/Computer%20Science/DailyForge/backend/src/main/java/com/dailyforge/modules/plan/application/service/CycleTemplateQueryApplicationService.java)
- 输出组装：
  - [CycleTemplateAssembler.java](/D:/Computer%20Science/DailyForge/backend/src/main/java/com/dailyforge/modules/plan/application/assembler/CycleTemplateAssembler.java)

### 2.3 当前动作元数据不足以支撑“自动带出默认结构”

现在后端读取动作时，`ExerciseReadEntity` 只读了这些字段：

- `id`
- `ownerUserId`
- `name`
- `isActive`

文件：

- [ExerciseReadEntity.java](/D:/Computer%20Science/DailyForge/backend/src/main/java/com/dailyforge/modules/plan/infrastructure/persistence/entity/ExerciseReadEntity.java)

但 `exercises` 表实际上还有：

- `exercise_type`
- `movement_type`
- `default_unit`

这意味着当前后端只能做“动作是否存在、是否是系统动作”的校验，做不了：

- 该动作默认是 `set_based` 还是 `single_segment`
- 该动作允许哪些参数
- 该动作推荐展示哪些参数

### 2.4 当前 `training_session` 表结构仍偏固定字段/组模型

虽然训练打卡模块还没正式开发，但表已经建好了，且明显偏向旧模型：

- `training_session_exercises.planned_snapshot_json`
- `training_session_exercises.actual_summary_json`
- `training_session_sets`

这套结构的问题是：

- `training_session_exercises` 用 JSON 保存计划和实际摘要，短期灵活，长期难校验、难查询
- `training_session_sets` 强绑定“按组训练”，不适配 `single_segment`
- 如果模板已经升级为“三层结构”，训练打卡还用“动作 + sets 表”的方式，后续映射会很拧巴

## 3. 这次改造在后端应该怎么实现

### 3.1 推荐采用“三层结构化模型”

我建议后端正式采用你草案里的三层抽象：

1. 动作
2. 执行项
3. 执行项参数

对应到数据库与 Java 模型：

- 动作头：`cycle_day_exercises`
- 执行项：`cycle_day_exercise_items`
- 执行项参数：`cycle_day_exercise_item_metrics`

这套方案比“把结构塞进一个 JSON 字段”更适合现在项目的原因：

- 现有 `cycle_template` 已经做了版本模型，继续结构化更自然
- 未来 `training_session` 明显也要复用这套结构
- 你后面要做统计和 AI，总会需要结构化查询能力

### 3.2 Java 侧建议新增统一领域概念

除了拆表，后端还要新增一套稳定的领域字典，不然结构会散在 Controller/Service 里。

建议新增：

- `StructureType`
  - `set_based`
  - `single_segment`
- `ItemType`
  - `set`
  - `segment`
- `MetricKey`
  - `weight_kg`
  - `reps`
  - `duration_seconds`
  - `distance_km`
  - `speed_kmh`
  - `pace_seconds_per_km`
  - `incline_percent`
  - `rest_seconds`
  - `rpe`
  - `intensity_level`
- `MetricValueType`
  - `number`
  - `text`

还需要一套“参数字典规则”，例如：

- 哪些参数允许出现在 `set_based`
- 哪些参数允许出现在 `single_segment`
- 参数值类型和范围
- 同一个执行项里某个 `metricKey` 是否允许重复
- `single_segment` 是否允许多个执行项

这些规则建议集中放在新的领域服务里，而不是散在 DTO 注解里。

### 3.3 动作默认结构的实现建议

要实现“选动作后自动带出默认结构”，后端必须知道动作的默认结构来源。

我建议有两个可选方案：

#### 方案 A：最省改动

直接基于 `exercises.exercise_type` 和 `default_unit` 在后端推断：

- 力量 / 器械 / 自重动作 -> `set_based`
- 跑步 / 椭圆机 / 爬坡 / 骑行 -> `single_segment`

优点：

- 不改 `exercises` 表
- 开发快

缺点：

- 规则隐含在代码里，不够显式
- 个别动作很难 override

#### 方案 B：更稳的长期方案

给 `exercises` 表新增明确元数据，例如：

- `default_structure_type`
- 可选：`metric_profile_code`

优点：

- 默认结构是数据，而不是硬编码
- 后续动作库维护更清晰

缺点：

- 需要额外迁移和种子数据更新

我的建议：

- 如果你准备很快做动作库筛选、动作推荐、AI 生成模板，选方案 B 更稳
- 如果你只是先把模板编辑器做出来，MVP 可以先用方案 A

## 4. 受影响的数据库结构

## 4.1 直接受影响的表

### 1. `cycle_day_exercises`

当前角色：

- 模板日下的动作表

建议改造：

- `id`

- `default_structure_type`
- 可选：`metric_profile_code`

如果不想加表字段，至少也要扩展 `ExerciseReadMapper` 查询：

- `exercise_type`
- `movement_type`
- `default_unit`

## 4.2 间接受影响、建议同步重构的表

### 1. `training_session_exercises`

当前问题：

- `planned_snapshot_json` / `actual_summary_json` 太自由
- 缺少“执行项头”和“参数头”

建议方向：

- 保留 `training_session_exercises` 作为打卡动作头表
- 新增：
  - `structure_type`
- 未来再新增：
  - `training_session_exercise_items`
  - `training_session_exercise_item_metrics`

### 2. `training_session_sets`

这是本次改造里最需要警惕的一张表。

因为它默认假设：

- 所有动作都能按 `set_no`
- 所有记录都能落成 `planned_weight_kg / actual_weight_kg / planned_reps / actual_reps`

这和 `single_segment` 明显冲突。

我的建议是：

- 不要再继续围绕 `training_session_sets` 追加业务代码
- 如果你决定推进这次动作参数模型升级，`training_session_sets` 应该视为待替换模型

## 5. 受影响的接口

## 5.1 当前 `cycle_template` 已有接口中，直接变更契约的接口

### 1. `POST /api/cycle-templates/drafts`

当前：

- `days[].exercises[]` 里是固定字段

改造后：

- `days[].exercises[]` 需要改为：
  - `exerciseId`
  - `structureType`
  - `note`
  - `items[]`

### 2. `PUT /api/cycle-templates/drafts/{templateId}`

同上，请求体会整体升级为新结构。

### 3. `PUT /api/cycle-templates/{templateId}`

同上，正式模板编辑也要改新结构。

### 4. `GET /api/cycle-templates/{templateId}`

这是查询接口里受影响最大的一个。

当前返回：

- `targetSets`
- `targetRepsMin`
- `targetRepsMax`
- `targetWeightKg`
- `targetDurationSeconds`
- `restSeconds`
- `targetRpe`
- `targetExtraJson`

改造后要返回：

- `structureType`
- `note`
- `items[]`
  - `itemIndex`
  - `itemType`
  - `itemName`
  - `metrics[]`

### 5. `POST /api/cycle-templates/{templateId}/copy`

外部接口契约可以不变，但内部复制逻辑必须改：

- 现在只复制 `cycle_day_exercises`
- 改造后需要连带复制 items 和 metrics

### 6. `POST /api/cycle-templates/{templateId}/activate`

外部响应可以不变，但启用前校验会变：

- 不能只校验 `cycleLength`
- 还要校验动作结构是否合法

## 5.2 当前 `cycle_template` 接口中，外部契约可以不变的接口

这些接口理论上可以不改外部返回结构：

- `GET /api/cycle-templates/formal`
- `GET /api/cycle-templates/drafts`
- `GET /api/cycle-templates/active/current`
- `DELETE /api/cycle-templates/{templateId}`

但它们的内部查询/统计逻辑可能仍会受影响，例如：

- `configuredDayCount` 的定义是否继续仅按 day 数计算
- 后续列表是否要展示结构摘要

## 5.3 未来 `training_session` 接口会被连带影响

虽然当前还没写，但如果这次改造成立，训练打卡接口应当从一开始就按新结构设计：

- 训练开始拉取计划快照
- 提交动作完成情况
- 查看历史训练详情

都应该统一使用：

- 动作
- 执行项
- 执行项参数

而不是新模板配老打卡结构。

## 7. 推荐的数据库迁移策略

## 7.1 推荐走“两阶段迁移”

### 阶段 A：兼容期迁移

新增不删：

- `cycle_day_exercises.structure_type`
- `cycle_day_exercise_items`
- `cycle_day_exercise_item_metrics`

保留旧固定字段，先不删。

后端策略：

- 新接口只写新结构
- 查询优先读新结构
- 如果某条老数据还没有新结构，可做一次兼容映射

### 阶段 B：收口期迁移

等前端和测试都切到新结构后，再删除旧字段：

- `target_sets`
- `target_reps_min`
- `target_reps_max`
- `target_weight_kg`
- `target_duration_seconds`
- `target_rest_seconds`
- `target_rpe`
- `target_extra_json`

## 7.2 如果你确认可以接受破坏式升级

如果满足以下条件：

- 当前没有需要保留的正式模板数据
- 前后端会一起改
- 没有外部用户在用旧接口

那也可以直接一刀切：

- 新增三层表
- 改 DTO / VO
- 同步删老字段

优点：

- 实现更干净
- 不用双读双写

缺点：

- 风险是一次性切换更大
- 回退成本更高

我的建议：

- 从工程稳妥性上，优先推荐“两阶段迁移”
- 从你现在项目阶段看，如果你确认本地没有要保的模板数据，也可以考虑“一刀切”

## 8. 一个很重要的迁移难点

## 8.1 现有“组数 + 次数范围”无法无损映射到草案里的 MVP 参数集

这是我在读草案和现有代码时发现的最大设计冲突。

当前模型支持：

- `target_sets = 4`
- `target_reps_min = 8`
- `target_reps_max = 10`

但草案里的 MVP 参数集只有：

- `reps`

没有：

- `reps_min`
- `reps_max`

这会带来两个问题：

1. 现有模板里常见的 `8~10 次` 没法自然表达
2. 如果只保留 `reps`，力量训练模板会被迫从“范围目标”降级成“精确目标”

### 8.2 这个问题不解决，会直接影响后端模型设计

如果你决定继续支持“次数范围”，那后端需要至少二选一：

#### 方案 A

把参数字典补成：

- `reps_min`
- `reps_max`

同理如果以后需要，也可以有：

- `duration_min_seconds`
- `duration_max_seconds`

#### 方案 B

把“范围”抽象成参数值结构，但这会明显提高模型复杂度，不建议这轮做。

我的建议：

- MVP 继续保留“范围型目标”能力
- 直接在参数字典中加入 `reps_min` / `reps_max`

这比强行把所有目标都改成精确值更贴近当前健身计划习惯。

## 9. 推荐的后端落地顺序

## 11. 最终建议

我的最终建议是：

2. 后端上直接按“三层结构化模型”设计，不要走“DTO 嵌套 + 单 JSON 落库”。

4. 先确认 4 个关键决策，再开始写 DDL 和新版接口文档。

这 4 个关键决策是：

1. `8~10 次` 这种范围型目标，是否继续支持  
   我建议：支持，并补 `reps_min/reps_max`
2. 默认结构类型来源于哪里  
   我建议：长期走动作表显式字段，MVP 可先代码推断
3. `single_segment` 在 MVP 是否只允许 1 个执行项  
   我建议：是，先限制成 1 个
4. 本次是否接受破坏式迁移  
   我建议：如果没有要保的模板数据，可以接受；否则走两阶段迁移
