## 1. 文档目标

本文档用于把“动作弹窗选择器”相关的后端改造范围收口成一份可直接执行的开发清单，供后端 agent 按此实现。

本次改造目标不是重做动作模块，而是在现有 `exercise` 查询能力上补齐：

- 动作选择器筛选元数据接口
- 粗粒度分类筛选能力
- 列表项肌肉信息增强

---

## 2. 已确认结论

1. 当前数据库结构足够，本次不改表。
2. 当前必须新增一个“筛选元数据接口”，避免前端硬编码左侧分类。
3. 当前列表接口需要新增 `categoryCode`，用于粗粒度分类过滤。
4. 当前列表接口里的 `primaryMuscles` / `secondaryMuscles` 不再只返回名称数组，升级为轻量对象数组。
5. 当前 `muscleId` 保留，但语义明确为“精确匹配某个肌肉节点”，不是自动包含子节点。
6. 一级分类是产品层概念，不要求与 `muscles` 表中的 `group` 节点完全一一对应。

## 5.1 接口层改造

### 新增接口

新增：

- `GET /api/exercises/system/filter-options`

用途：

- 返回动作选择器左侧一级分类
- 返回各一级分类下可选细分肌肉列表

建议放置位置：

- [ExerciseController.java](/D:/Computer%20Science/DailyForge/backend/src/main/java/com/dailyforge/modules/exercise/interfaces/rest/ExerciseController.java)

### 扩展现有列表接口

改造：

- `GET /api/exercises/system`

新增查询参数：

- `categoryCode`

保留查询参数：

- `keyword`
- `exerciseType`
- `movementType`
- `structureType`
- `sceneType`
- `muscleId`
- `page`
- `pageSize`

参数语义要求：

- `categoryCode`
  - 面向一级分类过滤
  - 示例值：
    - `chest`
    - `back`
    - `shoulder`
    - `arms`
    - `legs`
    - `core`
    - `cardio`
- `muscleId`
  - 面向精确肌肉过滤
  - 不负责自动展开子肌肉

## 5.4 查询与 Mapper 改造

### `ExerciseQueryMapper`

文件：

- [ExerciseQueryMapper.java](/D:/Computer%20Science/DailyForge/backend/src/main/java/com/dailyforge/modules/exercise/infrastructure/persistence/mapper/ExerciseQueryMapper.java)

需要改造：

1. 列表查询 SQL 支持 `categoryCode`
2. 不能直接在 SQL 里把 `categoryCode` 当数据库字段查
3. 正确做法是先在应用层把 `categoryCode` 转成一组肌肉 ID，再传给 SQL

建议实现方式：

- `categoryCode -> List<Long> muscleIds`
- SQL 侧支持：
  - `muscleId`
  - `muscleIds`

过滤语义建议：

- `categoryCode` 命中时：
  - 动作只要关联到该分类映射的任意肌肉，即视为命中
- `muscleId` 命中时：
  - 动作必须精确关联该肌肉
- 两者同时传入时：
  - 按“与”关系处理

### 新增肌肉树读取 Mapper

当前如果没有专门读取 `muscles` 树的 Mapper，建议新增：

- `ExerciseMuscleTreeQueryMapper`

职责：

- 按启用状态读取肌肉节点
- 返回：
  - `id`
  - `name`
  - `code`
  - `parent_id`
  - `muscle_level`
  - `sort_order`

说明：

- 这份读取能力主要服务 `filter-options`
- 不建议让 `ExerciseQueryMapper` 同时承担列表查询和肌肉树查询两类职责

## 5.5 分类映射策略

本次必须明确一层“产品分类到肌肉集合”的映射规则。

建议当前先在后端代码中维护固定映射，不落库。

推荐一级分类：

1. `chest`
2. `back`
3. `shoulder`
4. `arms`
5. `legs`
6. `core`
7. `cardio`

推荐映射思路：

- `chest`
  - 胸大肌及其下属细分肌肉
- `back`
  - 背部及其下属细分肌肉
- `shoulder`
  - 三角肌及其下属细分肌肉
- `arms`
  - 肱二头肌、肱三头肌、前臂
- `legs`
  - 臀部、腿部、小腿及其下属细分肌肉
- `core`
  - 核心及其下属细分肌肉
- `cardio`
  - 心肺

实现要求：

1. 不要在前端硬编码这套映射。
2. 不要要求前端传多个 `muscleId` 自己拼。
3. 后端应负责把分类映射收口成稳定契约。

## 7. 交付验收标准

后端完成后，应满足以下条件：

1. 前端可以通过一个接口拿到左侧分类和二级肌肉列表。
2. 前端点击“胸/背/肩/腿/手臂/核心/有氧”时，不需要自己维护肌肉 ID 映射。
3. 列表接口支持 `categoryCode`。
4. 列表接口返回的肌肉信息可直接用于前端标签展示和后续扩展。
5. 不需要修改现有数据库表结构。
